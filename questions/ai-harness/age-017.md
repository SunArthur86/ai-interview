# 蓝绿与金丝雀如何取舍

蓝绿部署与金丝雀发布是两种主要的发布策略，取舍取决于业务场景与资源约束：

**1. 蓝绿部署**
*   **原理**：维护两套完全相同的生产环境（蓝和绿）。在同一时刻，只有一套环境承载流量。切换时通过路由将流量瞬间从蓝切到绿。
*   **优点**：
    *   **回滚极快**：只需切回路由，秒级回滚，无代码回滚。
    *   **环境隔离**：新版本在无流量情况下预热，完全就绪后切换。
    *   **无版本共存**：避免了数据 Schema 冲突（在切换瞬间完成）。
*   **缺点**：
    *   **资源冗余**：需要双倍的服务器资源（2x Cost）。
    *   **停机感知（非平滑）**：虽然有连接排空，但长连接（WebSocket）可能受影响。
*   **适用场景**：对稳定性要求极高、可接受高成本、需要快速回滚的传统后端服务。

**2. 金丝雀发布**
*   **原理**：在生产环境中保留旧版本，部署新版本，通过灰度规则（如 5% 流量）逐步将用户引流到新版本。
*   **优点**：
    *   **资源利用率高**：只需逐步扩容新版本，最终替换旧版本，资源占用平滑。
    *   **精准验证**：可以针对特定用户（如内部员工）先发布，收集反馈。
    *   **错误受控**：如果新版本报错，只有小部分用户受影响，影响面可控。
*   **缺点**：
    *   **配置复杂**：需要精细的流量控制网关（如 Istio, Nginx）。
    *   **回滚稍慢**：需要逐步调整流量比例至 0，不如蓝绿切换干脆。
    *   **数据兼容挑战**：新旧版本同时运行，必须保证数据库 Schema 的向前/向后兼容性。
*   **适用场景**：资源敏感、高频迭代、需要验证业务逻辑或用户体验的服务（如推荐算法、前端页面）。

### 实战对比

| 特性 | 蓝绿部署 | 金丝雀发布 |
| :--- | :--- | :--- |
| **资源消耗** | 高 (需要全量备用环境) | 低 (仅需少量实例起步) |
| **回滚速度** | 秒级 (路由切换) | 分钟级 (流量逐步调整) |
| **流量控制** | 粗粒度 (全量切换) | 精细粒度 (百分比/用户特征) |
| **发布时长** | 快速 (部署后即刻切换) | 较慢 (需逐步观察放量) |
| **复杂度** | 低 (基础设施层支持即可) | 高 (需网关/Service Mesh支持) |

### 实战案例
在一次支付网关升级中，我们采用了**金丝雀发布**。起初仅 1% 流量，但新版本引入的了一个第三方库导致偶发的死锁，只影响了极少量用户。我们立即通过流量权重回滚，避免了全站资损。如果是蓝绿部署，一旦切流全量，故障面将是 100%，且回滚需重新切回，风险远高于金丝雀。

### 代码示例 (Nginx 权重配置)
```nginx
upstream backend {
    # 金丝雀：新版本 V2 占 10% 流量
    server new_v2_pod:8080 weight=1;
    # 旧版本 V1 占 90% 流量
    server old_v1_pod:8080 weight=9;
}
```

## 常见考点
1.  **在微服务架构中，金丝雀发布如何保证全链路一致性？**（追问：服务 A 调用服务 B，A 升级了 B 没升级怎么办？——Trace 驱动的灰度或 Header 透传）
2.  **数据库变更在蓝绿/金丝雀中如何处理？**（追问：先改库还是先发代码？——兼容性设计，Add Column, Rename Column 双写等）
3.  **K8s 中如何实现这两种策略？**（追问：Service Selector 切换 vs Ingress 权重调整）
