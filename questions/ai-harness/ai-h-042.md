# LLM应用如何设计高可用架构?LLM API不稳定时如何保证服务

- **高可用架构设计:**

- **1. 多供应商冗余**
```
用户请求 → API网关
├─ 主供应商(OpenAI) ← 健康检查
├─ 备供应商(Anthropic) ← 自动切换
└─ 兜底(本地模型/Ollama) ← 极端情况
```

- **2. 请求级容错**
```python
async def call_llm_with_failover(prompt):
providers = [
('openai', 'gpt-4', 0.7),      # 70%流量
('anthropic', 'claude-3.5', 0.2), # 20%
('zhipu', 'glm-4', 0.1),       # 10%
]
for provider, model, weight in providers:
**try：** 
return await call(provider, model, prompt)
except (Timeout, RateLimit, ServerError):
continue
# 所有供应商都失败
return fallback_response
```

- **3. 缓存兜底**
- 热门查询缓存结果
- API全部不可用时返回缓存

- **4. 降级策略**
| 级别 | 策略 |
|------|------|
| L0 | 正常服务 |
| L1 | 切换备用供应商 |
| L2 | 使用小模型(更快) |
| L3 | 返回缓存结果 |
| L4 | 返回静态回复(维护中) |

- **SLA目标:** 99.9%可用性 = 每月最多43分钟不可用
