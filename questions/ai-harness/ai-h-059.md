# LLM 应用的监控和传统 Web 应用的监控有什么不同

- **LLM 监控的独特维度**

传统 Web:CPU/内存/QPS/延迟
LLM 应用还需要:

1. **Token 消耗**:每次请求消耗的 token 数(直接关联成本)
2. **质量指标**:幻觉率、回答相关性、用户满意度
3. **LLM 延迟**:首 token 延迟 (TTFT) + 每秒生成速度
4. **工具调用成功率**:Agent 场景中工具调用的成功/失败率
5. **Prompt 变更追踪**:不同 prompt 版本的效果对比

- **常用工具**:LangSmith, Phoenix (Arize), Langfuse, W&B

- **一句话理解**:传统监控关注'系统是否正常',LLM 监控还要关注'AI 回答是否靠谱'.

- **LLM 可观测性数据流**
```
用户请求
    │
    ▼
┌───────────────┐      ┌───────────────┐
│  Tracing Span │─────▶│ Trace Collection│
│ (Prompt/Ans) │      │   (LangSmith) │
└───────┬───────┘      └───────┬───────┘
        │                    │
        ▼                    ▼
┌───────────────┐      ┌───────────────┐
│   Metrics     │      │   Evaluation  │
│ (Latency/Cost)│─────▶│ (Quality/Bleu)│
└───────────────┘      └───────────────┘
```

- **## 常见考点**
1. **TTFT vs TPOT**: 解释首 Token 延迟和每 Token 生成时间的区别及其对用户体验的影响。
2. **Tracing 实现**: 在 LangChain 中如何将应用日志自动上传到 LangSmith？（配置 LANGCHAIN_TRACING_V2 环境变量和 API Key）
3. **评估指标**: 如何自动化评估 LLM 的回答质量？（使用 LLM-as-a-Judge，让 GPT-4 给回答打分，或计算 Embedding 相似度）
