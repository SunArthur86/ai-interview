# 如何构建一套 AI Agent 的回归测试体系？核心难点在哪里？

Agent 的回归测试旨在确保在代码或模型更新后，原有的工具调用链路和逻辑依然正确。由于 Agent 的非确定性（随机采样）和外部依赖（API、环境），测试难度远超传统软件。

**构建体系**：
1.  **场景库建设**：收集真实的用户 Query 和对应的最优工具调用序列（Golden Trace）。
2.  **可重复性模拟**：构建 Mock Server，模拟外部工具的返回结果，确保测试环境不受外部 API 变动影响。
3.  **评估指标**：除了最终答案的正确性（Exact Match 或 LLM-as-a-Judge），还要检查中间过程是否调用了正确的工具以及参数是否合规。
4.  **确定性控制**：在测试时固定 Seed 和 Temperature，减少随机性波动。

**核心难点**：
1.  **非确定性输出**：即使 Seed 固定，底层框架或模型版本的微小差异也可能导致输出路径改变。
2.  **评估成本高**：需要真实运行多步推理和外部调用，测试耗时且消耗 Token。
3.  **长尾场景覆盖**：Agent 面对复杂异常时的行为难以穷举。

### 测试闭环流程图

              ┌───────────────────────┐  ┌─────────────────┐

                           │                        │                        │

### 实战案例
某订票 Agent 在回归测试中经常失败，原因是天气 API 的实时数据变动导致 Agent 无法复现历史决策。通过引入 **VCR (Video Cassette Recorder)** 模式录制真实的 API 响应并在测试时回放，将测试通过率从 60% 稳定提升至 95% 以上。

### Agent 测试方法对比
| 维度 | Golden Trace (录制回放) | LLM-as-a-Judge (模糊评估) | 传统单元测试 (Mock) |
| :--- | :--- | :--- | :--- |
| **确定性** | 高（严格匹配路径） | 中（依赖 Judge 模型一致性） | 极高（逻辑固定）
| **覆盖范围** | 覆盖已知路径 | 可覆盖未知/边缘场景 | 覆盖特定代码分支 |
| **维护成本** | 高（业务变更需更新录制） | 中（需编写 Prompt 和标准） | 低（代码级 Mock）
| **执行速度** | 快（无外部 I/O） | 慢（需调用 Judge 模型） | 极快（仅本地执行）
| **适用阶段** | 回归测试、CI/CD | 离线评估、模型迭代 | 开发阶段、逻辑调试 |

### 代码示例 (Pytest - Mock 工具调用)
```python
import pytest
from my_agent import Agent
from unittest.mock import patch, MagicMock

# 定义测试用例：查询天气
@pytest.mark.asyncio
async def test_weather_tool_call():
    # 1. 准备 Mock 返回数据
    mock_weather_response = {"temperature": 25, "condition": "sunny"}
    
    # 2. Patch 外部 API 调用
    with patch('my_agent.tools.get_weather_api', return_value=mock_weather_response) as mock_api:
        agent = Agent()
        user_query = "今天北京天气怎么样？"
        
        # 3. 执行 Agent
        result = await agent.run(user_query)
        
        # 4. 断言：验证是否调用了正确的工具且参数正确
        mock_api.assert_called_once_with(city="北京")
        # 5. 断言：验证最终答案包含关键信息
        assert "25" in result.output and "sunny" in result.output
```
