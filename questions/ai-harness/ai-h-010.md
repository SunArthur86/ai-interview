# LLM框架本身的性能开销有多大?如何选择性能最优的框架

- **框架开销来源:**
1. **抽象层开销** - 额外的类/函数调用
2. **序列化/反序列化** - Prompt构造/输出解析
3. **依赖膨胀** - 安装包大、import慢
4. **错误处理** - 重试/fallback逻辑

- **性能对比(简单RAG任务基准测试):**

| 方案 | 延迟(ms) | 内存(MB) | 依赖数 |
|------|---------|---------|--------|
| 原生API | **~50** | **~20** | **1** |
| LangChain | ~200 | ~150 | 200+ |
| LlamaIndex | ~180 | ~120 | 100+ |
| Haystack | ~160 | ~100 | 80+ |

- **性能优化建议:**

1. **延迟敏感 → 原生API**
- 去掉框架中间层
- 手动管理prompt和解析

2. **复杂逻辑 → 框架但精简依赖**
```python
# 只安装需要的
pip install langchain-core  # 而非 langchain
```

3. **批量处理 → 减少框架开销**
- 用batch API而非循环单次调用
- 异步并发(asyncio)

4. **缓存层**
- Prompt缓存(系统prompt复用)
- 结果缓存(相同输入直接返回)

- **结论:** 对于性能关键场景,推荐原生API + 轻量辅助库(如Instructor for structured output)
