# 如何监控一次 Agent 任务的「真实成本」

### 核心逻辑：精细化账单 = (模型推理 + 外部调用 + 基础设施) × 归因

#### 1. 模型推理成本（核心）
- **计算公式**：`Σ (Prompt Tokens × Input Price) + Σ (Completion Tokens × Output Price)`
- **关键细节**：
  - 需区分不同模型的定价（如 GPT-4o vs. GPT-3.5-Turbo）。
  - 注意 Token 计数方式，需使用各模型官方提供的 Tokenizer（如 `cl100k_base`）精确计算，而非简单的字符/单词估算，避免计费偏差。
  - 需捕获流式输出中的实际 Usage 字段，而非预设上限。

#### 2. 外部调用与检索成本
- **向量数据库**：计算读取请求的 Read Units (RU) 和写入的 Write Units (WU) 费用。
- **API 工具**：如调用搜索 API (Tavily/Bing)、天气 API 等的按次调用费用。
- **函数调用**：若 Agent 调用自身托管的长耗时工具，需计算 GPU/CPU 实例时长。

#### 3. 基础设施与固定成本
- **计算资源**：Agent 运行时所在的 Pod/容器资源（CPU/内存/GPU）小时费。
- **网络带宽**：出入站流量费（对于大量上下文传输不可忽视）。
- **存储**：向量库存储、对话历史存储（S3/EBS）的费用。

#### 4. 归因与报表（成本分摊）
- **Trace ID 关联**：通过 TraceID 将上述所有环节串联。
- **多维报表**：按租户、用户、具体的 Agent 链路（如“客服机器人”vs“代码助手”）进行聚合。

#### 5. 预算告警与控制
- **硬限制**：当租户预算耗尽，在网关层直接拦截请求。
- **软限制**：达到阈值（如 80%）触发邮件/Slack 告警，或自动降级模型（从 GPT-4 切换到 GPT-3.5）。

## 常见考点
1. **如何处理流式输出的 Token 计算**？
   - 答：不能只数 Prompt，必须在响应流结束或进行中的 `usage` 字段获取 `completion_tokens`，并确保累计计算。
2. **隐藏成本在哪里**？
   - 答：往往被忽视的是 Context Buffering（长上下文重新输入）导致的重复计费，以及向量库的高频查询 RU 费用。
3. **如何实现实时成本控制**？
   - 答：在请求发起前根据 Prompt 预估 Token × Price，若超预算则直接拦截；或在流式输出过程中监控 Token 达到上限强制截断。
