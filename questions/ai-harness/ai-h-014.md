# 如何处理RAG中的表格、图表、公式等非文本内容

- **非文本内容RAG方案:**

1. **表格处理**
- 提取为Markdown/CSV格式
- 用LLM生成表格摘要
- 存储为:摘要(embedding) + 原始表格(精确返回)
- 查询时:摘要检索 → 原始表格注入Prompt

2. **图表/图片处理**
- VLM(如GPT-4V/LLaVA)生成图片描述
- 描述做embedding
- 原图URL/路径存储

3. **公式处理**
- MathPix API转换为LaTeX
- LaTeX文本做embedding
- 渲染后的公式图片用VLM描述

4. **PDF复杂版面**
- 使用Unstructured.io/Marker解析
- 保留页面结构(标题/段落/表格/图)
- 带页面号的metadata

- **多模态RAG架构:**
```
文档 → 解析器(文本/表格/图)
→ 分别处理(文本embedding/VLM描述/表格摘要)
→ 统一向量索引
查询 → 检索 → 多模态结果聚合 → LLM生成
```

**多模态 RAG 数据处理流水线:**
```text
[原始文档 PDF]
│
├─> [Unstructured / Marker 解析]
│    ├─ Text Block ──────────────┐
│    ├─ Table Block ──> LLM ────┤→ Embedding Model ──> [向量 DB]
│    │              (Summary)    │
│    └─ Image Block ──> VLM ────┤
│                   (Caption)    │
│                              │
└─> [原内容存储] ────────────────┘ (图片存 S3, 表格存 JSON)
```

## 常见考点
1. **Table Parsing 难点**：解析多级表头或合并单元格时，常规 OCR/PDF 解析器容易出错，如何解决？（答：使用专门的表格解析模型如 Table-Transformer 或对表格转图片后用 VLM 理解）
2. **多模态 Embedding**：为什么不直接用 CLIP 等多模态模型对图片做 Embedding 存入向量库？（答：可以，但目前的图文对齐模型在精细信息（如图表坐标）理解上不如文本 VLM 描述准确）
3. **分辨率限制**：VLM 输入图片的分辨率限制会影响大图表的读取，如何处理？（答：对大图进行切片，分块描述后再合并，或使用支持高分辨率的模型）
