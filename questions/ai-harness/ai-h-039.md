# 模型量化方案GGUF vs AWQ vs GPTQ有什么区别?如何选择

- **PagedAttention:受OS虚拟内存启发的KV Cache管理**

- **传统KV Cache问题:**
- **静态分配**: 每个请求预分配连续且较大的内存块（如预设2048 tokens）。
- **内存浪费**: 实际输出往往较短（如512 tokens），剩余预分配内存闲置。
- **内部碎片**: 即使总空闲内存足够，但无法容纳新的长序列请求，因为必须连续。
- **低GPU利用率**: 内存碎片化导致显存无法有效利用，GPU利用率低(30-60%)。

- **PagedAttention解决方案:**

1. **分页管理**
- 将KV Cache切成固定大小的Block（如每Block存16个Token的KV）。
- 物理显存中的Block可以不连续存储。
- 维护一个Block表（类似OS页表），映射逻辑序列索引到物理Block。

2. **按需分配 (Non-contiguous Allocation)**
- 生成新Token时，逐块申请物理内存。
- 不需要大块连续显存，消除内部碎片。

3. **共享 (KV Cache Sharing)**
- 不同的Prompt请求如果拥有相同的前缀（如System Prompt），可以共享计算好的KV Cache Block。
- 修改时才复制，极大节省显存。

- **架构对比图:**
```
传统方式:
Request 1: [████████████████████░░░░] (浪费)
Request 2: [████░░░░░░░░░░░░░░░░░░░] (碎片)
显存: [████████████████████░░░░░░░░░░░░░░░░░░░] (无法插入新的大请求)

PagedAttention:
Block Table (Logical): [P1][P2][P3][P4] -> ... -> [P10]
Physical Memory:     [C1][C3][C2][C4][C5][C8]... (无需连续)
                    ^Share with Req B (C1-C2)
```

- **效果:**
- GPU显存利用率: 30-60% → **90%+**
- 吞吐量:** 2-4倍提升** (尤其是长文本和高并发场景)
- 内存浪费: <4%

- **Continuous Batching (Continuous/Iterative Batching):**
- **问题**: 静态Batching中，必须等最慢的请求生成完才能释放整个Batch，导致算力闲置。
- **解决**: vLLM将请求按Token维度动态调度。
  - 不同请求（不同生成长度）可以在同一个Batch中处理。
  - 请求A完成后，立即在Batch中加入请求C，无需等待。
  - 实现了精确到Token步数的Batch调度。

- **实战案例**：在部署长文档摘要服务（Context Length 32k）时，使用HuggingFace Transformers推理，单卡A100并发数只能达到2。迁移至vLLM后，利用PagedAttention解决了显存碎片问题，在相同的显存预算下并发数提升至16，且P95延迟降低了50%。

- **代码示例**：
```python
from vllm import LLM, SamplingParams

# 初始化 LLM 引擎 (自动启用 PagedAttention 和 Continuous Batching)
llm = LLM(model="meta-llama/Llama-2-7b-hf", 
          tensor_parallel_size=1, 
          block_size=16, # 每个 block 存 16 个 token 的 KV
          max_num_seqs=256) # 最大并发序列数

# 设置采样参数
prompts = ["你好，请介绍一下...", "写一段Python代码..."]
sampling_params = SamplingParams(temperature=0.7, top_p=0.95, max_tokens=100)

# 执行推理 (内部自动处理 Batch 调度和 KV 内存)
outputs = llm.generate(prompts, sampling_params)

for output in outputs:
    print(f"Output: {output.outputs[0].text}")
```

## 常见考点
1. **Block Size的选择**：Block大小（如16个token）如何影响性能？太大会导致浪费，太小会增加Block Table的管理开销和内存寻址延迟。
2. **Prefix Caching的细节**：KV Cache共享在多轮对话中效果显著，但vLLM早期版本仅支持同一Batch内的共享，新版本（如v0.4+）如何实现跨Batch的全局共享？
3. **解码时的内存带宽瓶颈**：PagedAttention主要解决显存管理问题，但在LLM推理（尤其是解码阶段）往往受限于显存带宽而非计算算力，vLLM如何配合Kernel优化（如FlashAttention）来缓解这一瓶颈？
