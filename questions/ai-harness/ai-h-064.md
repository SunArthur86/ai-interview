# 设计一个生产级多模型网关:如何实现模型路由、熔断和自动降级

- **多模型网关架构**

```
用户请求
→ 路由决策(基于 query 复杂度/成本/延迟)
→ 模型调用(GPT-4 / Claude / GLM / 本地 7B)
→ 熔断检查(如果模型连续失败 → 降级)
→ 降级链(大模型 → 小模型 → 缓存 → 兜底回复)
→ 响应
```

- **1. 模型路由策略**
- **规则路由**:代码问题 → Code模型,简单FAQ → 小模型
- **LLM 路由**:用轻量模型(如 BERT)分类 query 难度
- **成本路由**:Token 预算 < 500 → 小模型,> 2000 → 大模型
- **延迟路由**:SLA 要求 < 1s → 本地模型

- **2. 熔断器设计(三态)**
- **Closed(正常)**:请求正常通过
- **Open(熔断)**:连续失败超阈值 → 直接拒绝(快速失败)
- **Half-Open(探测)**:放少量请求试探是否恢复

```python
class CircuitBreaker:
def __init__(self, threshold=5, timeout=60):
self.failures = 0
self.state = 'closed'

async def call(self, model, prompt):
if self.state == 'open':
return await self.fallback(prompt)
**try：** 
result = await model.generate(prompt)
self.failures = 0  # 重置
return result
except Exception:
self.failures += 1
if self.failures >= self.threshold:
self.state = 'open'
schedule_reset()  # 60s 后转 half-open
return await self.fallback(prompt)
```

- **3. 降级链**
- Level 1: GPT-4 → Claude(不同厂商 fallback)
- Level 2: Claude → 本地 7B 模型(云 → 本地)
- Level 3: 本地模型 → 语义缓存(历史相似回答)
- Level 4: 缓存 → 固定兜底回复('服务繁忙,请稍后重试')

- **4. 监控**
- 每个模型的 QPS / 延迟 / 成功率 / 成本
- 路由分布(多少去了大模型 vs 小模型)
- 熔断触发频率
