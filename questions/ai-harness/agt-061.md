# 模型路由与容错

### 模型路由与容错

**概念解释**：
- **模型路由**：像快递分拣中心，根据任务类型、成本预算、延迟要求等，将请求动态分发到最合适的模型（如 GPT-4、Claude、开源小模型），而非全量跑大模型。
- **容错**：当模型超时、报错或限流时，系统自动切换、重试或降级，保障服务高可用。

**原理详解**：
1. **多模型管理（统一抽象）**：
   - **Provider Adapter**：统一各厂商接口（如 OpenAI、Anthropic），封装 `chat()` 方法。
   - **配置中心**：维护模型能力矩阵（是否支持函数调用、长上下文）、价格、限额。

2. **调度策略**：
   - **成本优先**：满足质量阈值下选最便宜模型。
   - **延迟优先**：SLA 严格时选最快模型。
   - **任务分类**：代码生成用强模型，简单摘要用弱模型。

3. **三态熔断器**：
   - **Closed（闭合）**：正常请求，统计失败率。
   - **Open（断开）**：失败超阈值，直接拒绝并进入冷却期（防止雪崩）。
   - **Half-Open（半开）**：冷却结束，放行少量探测流量，成功则恢复，失败则继续断开。

4. **自动降级**：
   - **模型降级**：强模型失败时切至弱模型（或更快的模型）。
   - **功能降级**：关闭复杂工具调用，改为单轮问答。
   - **策略**：配合日志标记 `degraded=true`，必要时提示用户。

5. **重试与指数退避**：
   - 针对 429/503 错误，等待时间指数增长（`base * 2^attempt + jitter`），避免瞬时拥塞。
   - **不可重试**：参数错误 (400)、权限错误 (401/403) 直接失败。

## 常见考点
1. **如何设计模型路由的“成本/质量”评估机制？**
   - 通常基于 Embedding 相似度或语义分类器。对于每个请求，提取特征（关键词、意图分类），打上标签（如“复杂推理”、“简单提取”、“创意生成”），然后映射到配置好的路由表。更高级的做法是使用一个轻量级 LLM（或小模型）作为“裁判”，判断任务难度。
2. **多模型并发与瀑布流策略的区别？**
   - **并发**: 同时请求 GPT-4 和 Claude，谁先返回用谁（低延迟，高成本）。
   - **瀑布流**: 先请求便宜/快速模型，如果置信度低或失败，再请求昂贵模型（平衡成本与质量）。生产环境推荐使用带置信度阈值的瀑布流策略。
3. **如何避免“惊群效应”导致下游全挂？**
   - 当主 Provider 挂掉时，如果有大量请求瞬间打到 Backup Provider，容易导致 Backup 也崩溃。解决方法包括：在熔断器开启时，在客户端侧直接限流或拒绝部分请求，而不是全部转发；或者使用随机抖动的重试时间分散流量。
