# MCP (Model Context Protocol) 的三层架构分别是什么？它解决了什么痛点？

MCP (Model Context Protocol) 是一种开放标准，旨在连接 AI 模型与本地/远程数据源和工具。其三层架构设计如下：

1.  **传输层**：定义了 Client（如 Desktop App）和 Host（如 MCP Server）之间的通信方式，通常是基于 JSON-RPC 2.0 的标准，支持 STDIO 或 SSE (Server-Sent Events) 传输，解决了不同语言和环境的底层通信问题。
2.  **会话层**：管理连接的生命周期、初始化握手和资源隔离。它定义了如何建立连接、交换 Capabilities（能力列表），确保连接的稳定性和协议一致性。
3.  **能力层**：定义了具体的接口类型，包括 Prompts（提示词模板）、Resources（读取文件/数据）、Tools（执行函数）等。这是模型实际交互的 API 表面。

**解决的痛点**：在 MCP 之前，每个 AI 应用都需要为每个数据源（如 Google Drive、Slack、本地文件）编写特定的 Connector（插件），导致生态碎片化。MCP 提供了统一的“即插即用”标准，使得任何兼容 MCP 的模型都能无缝接入任何实现了 MCP 协议的数据源，极大地降低了连接成本。

### 实战案例
构建企业知识库助手时，直接让 LLM 访问数据库存在注入风险。通过实现一个 MCP Server 作为中间层，将 SQL 查询封装为 Tool，Client 端仅传递参数，不仅统一了不同数据源（PG, ES）的调用方式，还通过 Server 端统一实现了 RBAC 权限控制。

### MCP 协议分层对比
| 层级 | 核心组件 | 职责 | 常见技术/标准 | 实战价值 |
| :--- | :--- | :--- | :--- | :--- |
| **能力层** | Resources, Tools, Prompts | 定义“能做什么”（数据读写、工具调用） | JSON Schema | 标准化接口，复用逻辑 |
| **会话层** | Initialization, Capabilities | 定义“怎么交互”（握手、生命周期） | JSON-RPC 2.0 Methods | 保证状态同步，错误处理 |
| **传输层** | STDIO, SSE | 定义“数据怎么跑”（载体、模式） | Standard Streams, HTTP | 解耦语言依赖，支持本地/远程 |

### 代码示例 (TypeScript - 简单的 MCP Tool 定义)
```typescript
import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";

const server = new Server(
  { name: "my-server", version: "1.0.0" },
  { capabilities: { tools: {} } }

// 注册一个计算器工具
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  if (request.params.name === "calculate") {
    const args = request.params.arguments as { a: number; b: number };
    return { content: [{ type: "text", text: String(args.a + args.b) }] };
  }
});

// 启动传输层（通常用于本地进程通信）
const transport = new StdioServerTransport();
await server.connect(transport);
```
