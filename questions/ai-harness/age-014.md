# 语义缓存如何保证安全

语义缓存通过向量相似度复用历史结果，其安全性保障机制如下：

1.  **严格的租户隔离**：
    *   缓存 Key 必须包含 `TenantID`，防止跨租户数据泄露（Tenant A 的敏感回答被 Tenant B 命中）。
2.  **Key 的版本与参数绑定**：
    *   缓存 Key 需包含 `ModelID`、`Prompt Template` 的 Hash 值。若模型升级或 Prompt 修改，必须自动失效旧缓存，防止返回过时或逻辑不符的答案。
3.  **相似度阈值保守设置**：
    *   设定较高的余弦相似度阈值（如 > 0.95），防止近似但不等义的请求命中缓存（如「转账给张三」与「转账给李四」）。对于涉及权限操作或 PII 信息的请求，建议禁用缓存。
4.  **敏感任务策略**：
    *   金融、医疗或写操作（Tool Calling）请求，建议强制标记 `Cache-Control: no-store`，或要求模型对缓存结果进行二次确认。
5.  **TTL 与主动失效**：
    *   设置合理的 TTL；对于知识库更新频繁的场景，提供「主动失效」接口，在文档更新时清除相关语义缓存。

## 常见考点
1.  **如何平衡缓存命中率与数据一致性？**（追问：TTL 设置经验，或者监听数据库 Binlog 来失效缓存）
2.  **如果 Prompt 有细微变动（如 System Prompt 改了一个字），缓存会失效吗？**（追问：Prompt Hash 策略）
3.  **语义缓存的「相似度」如果误判了权限怎么办？**（追问：ACL 检查是在缓存前做还是缓存后做？必须先做权限检查）
