# Prompt 设计原则

### 2. Prompt 设计原则

#### 2.1 清晰性（Clear）
**概念解释**
用词准确、无歧义，避免「它、这个、上面」指代不明；关键术语与业务定义需写清楚。

**原理详解**
歧义会放大模型的随机性；清晰指令缩小「合法输出」的空间，提高一致性。模型是基于概率预测下一个 Token，模糊的指令会导致概率分布过于分散，引入随机噪声。

**正面示例**
```text
【角色】你是电商售后客服质检助手，只根据给定规则做分类，不编造订单信息。
【任务】阅读用户留言，输出唯一标签：退货 / 换货 / 物流查询 / 其他。
【上下文】
用户留言：{{user_message}}
（若提供）订单号：{{order_id}}
【格式】
仅输出一行 JSON：{"label":"...", "confidence":0.0-1.0, "reason":"一句话理由"}
【约束】
- 若信息不足以判断，label 填 "其他"，confidence 降低并在 reason 说明缺失信息。
- 不要输出 JSON 以外的任何文字。
```

**反面示例**
```text
把日期改一下格式。
```

#### 2.2 具体性（Specific）
**概念解释**
说明粒度（多长、几个要点）、判定标准（何为「高优先级」）、反例（什么情况算错误）。

**原理详解**
边界模糊会导致模型自行脑补规则，具体性通过正反例和量化指标锁定输出范围。这是为了解决模型对「常识」和「业务规则」的混淆，例如「总结」默认可能侧重主要信息，但业务可能需要包含「免责声明」等看似次要但关键的信息。

**正面示例**
```text
用三条要点总结下文，每条不超过 25 个汉字；不得引入文中未出现的人名或数字。
```

#### 2.3 结构化（Structured）
**概念解释**
使用标题、编号、XML 标签、Markdown 小节、分隔线，使模型与人类都易于解析「哪部分是规则、哪部分是数据」。

**原理详解**
标签化（如 `<document>...</document>`）能降低「把用户数据当指令」的风险，也便于多段材料拼接。Transformer 的注意力机制对特殊分隔符敏感，结构化的 Prompt 能帮助模型更好地建立长距离依赖，区分指令区与数据区。

**Prompt 结构架构图**
```text
┌──────────────────────────────────────────────────────────────┐
│ System Prompt / Meta Instructions (角色设定、全局目标)       │
├──────────────────────────────────────────────────────────────┤
│ Delimiters & Rules (分隔符、约束条件、输出格式定义)          │
│  e.g., "Use XML tags below", "Output JSON only"              │
├──────────────────────────────────────────────────────────────┤
│ Few-Shot Examples (少样本示例)                               │
│  ┌─ Example 1 ─┐  ┌─ Example 2 ─┐  ┌─ Example 3 ─┐          │
│  │ Input: ...   │  │ Input: ...   │  │ Input: ...   │          │
│  │ Output: ...  │  │ Output: ...  │  │ Output: ...  │          │
│  └─────────────┘  └─────────────┘  └─────────────┘          │
├──────────────────────────────────────────────────────────────┤
│ Current Input (当前待处理数据)                               │
│  {{user_input}} or <context>...</context>                    │
├──────────────────────────────────────────────────────────────┤
│ Trigger Cue (触发提示，如 "Answer:" 或 "### Response:")    │
└──────────────────────────────────────────────────────────────┘
```

**正面示例**
```text
<rules>
仅使用 <article> 中的事实作答；若 <article> 未提及则回答「文中未提及」。
</rules>
<article>
...正文...
</article>
```

#### 2.4 迭代优化（Iterative）
**概念解释**
第一版 Prompt 很少完美；应建立失败样本集 → 改 Prompt → 回归测试的闭环。

**原理详解**
与写代码类似：用例驱动；记录「模型常错类型」（漏约束、格式错、过度推断），针对性加反例说明或步骤化。Prompt 优化往往遵循「长文本效应」，即随着 Prompt 变长、细节变多，模型性能通常会提升。

**面试 Q&A**
Q：你如何迭代优化 Prompt？
A：
1. 固定评测集与评分标准。
2. 分类错误（理解错、知识错、格式错、安全错）。
3. 小步修改，一次改一个变量。
4. 记录版本与效果，避免「感觉变好」但无数据。

#### 2.5 综合正反面示例
**任务**：从用户邮件中提取「会议时间」和「参会人邮箱」。

**较差 Prompt**
```text
从这封邮件里提取会议时间和邮箱。
```

**较好 Prompt**
```text
从邮件中提取会议时间和参会人邮箱。
要求：
1. 时间输出为 ISO 8601 格式 (YYYY-MM-DD HH:mm)。
2. 邮箱仅提取 xxx@xxx.com 格式。
3. 若无相关信息，字段填 null。

邮件内容：
{{email_content}}

输出 JSON：{"time": "...", "emails": [...]}
```

## 常见考点
1. **CoT（Chain of Thought）适用场景**：遇到复杂推理、数学或逻辑任务时，如何让模型「一步步思考」？在 Prompt 中添加 "Let's think step by step"。
2. **Prompt 注入防御**：当用户输入包含恶意指令时，如何通过分隔符和角色设定防止模型执行用户指令？例如使用 "### Input:" 包裹用户数据，并明确告知忽略该区域内的指令。
3. **长上下文与注意力衰减**：当 Prompt 非常长时（如超过 32k token），模型对开头指令的记忆力可能下降，如何解决这个问题？（关键指令重复放在尾部或使用 System Prompt 加固）。
