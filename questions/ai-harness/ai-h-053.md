# Cursor 的核心技术是什么?它和 GitHub Copilot 有什么区别

- **Cursor** 是基于 VS Code fork 的 AI 原生 IDE,核心技术包括:

- **1. 全仓库上下文理解:**
- Codebase Indexing:将整个代码仓库索引为向量
- 检索时用 RAG 找到相关代码片段
- 支持 @file、@function、@web 精确引用

- **2. Agent 模式:**
- 类似 Claude Code 的 Agent Loop
- 可以读写文件、执行命令、运行测试
- 支持多文件编辑

- **3. Inline 补全 + Chat:**
- Tab 补全(类似 Copilot)
- Cmd+K 行内编辑
- 侧边栏 Chat 带全仓库上下文

**vs GitHub Copilot:**

| 维度 | Cursor | Copilot |
|------|--------|---------|
| 上下文 | 全仓库 RAG | 当前文件 + 少量上下文 |
| 编辑能力 | 多文件 Agent | 单行/片段补全 |
| 模型选择 | 多模型(Claude/GPT/自定义) | OpenAI 限定 |
| IDE | 独立 IDE | 插件 |
| Agent 能力 | 强(Composer/Agent)| 弱 |

Cursor 的核心优势是 Agent 级别的全仓库编辑能力.

**## 常见考点**
1. **Composer 工作原理**：Cursor 的 Composer（多文件编辑 Agent）是如何避免产生“幻觉性修改”的？考察对 Diff 应用的理解和“Apply Patch”流程。
2. **索引与检索策略**：Cursor 如何在本地处理大型 Monorepo（如数百万行代码）？询问是否使用了 HNSW 算法、分层索引或 AST 静态分析来提升检索准确度。
3. **隐私与安全**：代码库索引是上传到云端还是本地处理？如何处理企业级的代码隐私合规问题？（Cursor 目前支持部分本地索引）。
4. **上下文窗口利用**：即使有了 RAG，大模型仍受限于上下文窗口。Cursor 是如何将检索到的代码块高效地喂给模型而不丢失依赖关系的？（考察 Context Packing 策略）。
