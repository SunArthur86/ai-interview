# LLM应用如何实现多租户隔离和权限控制?RAG知识库如何做文档级权限

- **多租户隔离方案:**

| 层级 | 方案 | 示例 | 优缺点 |
|------|------|------|--------|
| 数据隔离 | 租户独立的向量索引 | tenant_a_index, tenant_b_index | **强隔离，性能好**；资源消耗大，管理复杂 |
| 元数据过滤 | 同一索引+metadata过滤 | filter: {tenant_id: 'a'} | **资源利用率高**；查询性能稍降，需严格配置权限 |
| 应用隔离 | 每租户独立部署 | Docker/K8s namespace | **最强隔离**；运维成本极高 |

- **文档级权限控制:**

1. **Metadata标注**
将权限信息固化在向量Payload中。
```json
{
  "content": "高管薪酬报告",
  "metadata": {
    "doc_id": "doc_123",
    "access_level": "executive",
    "department": "finance",
    "tenant_id": "company_a",
    "allowed_roles": ["admin", "finance_manager"]
  }
}
```

2. **检索时过滤**
在执行向量搜索前，强制注入用户权限过滤器，确保向量数据库只返回有权限的片段。
```python
# Qdrant 示例
results = qdrant.search(
    collection_name="docs",
    query_vector=embedding,
    query_filter={
        "must": [
            {"key": "tenant_id", "match": {"value": user.tenant_id}},
            # 必须满足用户权限级别 >= 文档级别
            {"key": "access_level", "match": {"any": user.permissions}}
        ]
    }
)
```

3. **RBAC + ABAC**
- RBAC (Role-Based Access Control): 基于角色（如Admin, Editor）
- ABAC (Attribute-Based Access Control): 基于属性（部门/项目/安全等级/密级）
- *关键*：向量检索层面只做预过滤，最终生成答案前，LLM最好再次校验（通过System Prompt），防止检索逃逸。

## 常见考点
1. **预过滤性能影响**：在大规模向量检索中，复杂的Filter条件（如`must_not`或复杂的`any`逻辑）会显著降低HNSW图的遍历效率，如何优化？
2. **权限继承**：如果文档权限发生变更，向量库中的Metadata如何高效批量更新？
3. **Row-Level Security vs Data Level Security**：在RAG场景下，如何处理用户只能看到文档中部分段落（如合同中的脱敏条款）的极端权限需求？
