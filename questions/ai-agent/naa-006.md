# 设计一个生产级 Agent 编排系统,你会如何选择状态机 vs DAG vs 事件驱动

- **三种编排范式深度对比**

- **1. 状态机**
- 适用:明确的有限状态集 + 状态转换规则
- 优势:可枚举所有合法状态,便于 debug 和异常处理
- 劣势:状态爆炸(状态数随业务复杂度指数增长)
- 框架:LangGraph (StateGraph)
- 适合:对话管理、审批流程

- **2. DAG (有向无环图)**
- 适用:有明确依赖关系的任务编排
- 优势:支持并行子任务(同时查 DB 和调 API)
- 劣势:不支持循环和条件跳转(需要扩展为 conditional DAG)
- 框架:Airflow, Prefect
- 适合:数据处理管道、RAG 多路检索

- **3. 事件驱动**
- 适用:异步、解耦的微服务架构
- 优势:组件完全解耦,支持动态扩展
- 劣势:调试困难(事件流追踪难)
- 框架:Kafka + Event Sourcing
- 适合:实时监控、IoT 场景

- **生产级架构选择决策树**
```
任务是否有明确的有限状态?
├─ 是 → 状态数 < 20? → 状态机
│        └─ 否 → DAG + 条件分支
└─ 否 → 需要实时响应? → 事件驱动
└─ 否 → DAG
```

- **最佳实践:混合架构**
- 外层:状态机(管理对话状态)
- 中层:DAG(并行工具编排)
- 内层:事件驱动(异步日志/监控)
- 这是 ragent 等企业级 Agent 平台的实际架构
