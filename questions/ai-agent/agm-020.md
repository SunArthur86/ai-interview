# Agent记忆的Token预算如何分配才不容易翻车?不同记忆类型的Token配比建议

- **Token预算分配框架:**

假设总上下文128K tokens:

```
System Prompt:      2K  (1.5%)
**工具定义：** 3K  (2.5%)
**长期记忆检索：** 10K  (8%)
对话历史(近期):    15K  (12%)
**RAG文档检索：** 30K  (24%)
**当前任务输入：** 5K  (4%)
**输出预留：** 4K  (3%)
────────────────────────
**已用：** 69K / 128K
**剩余空间：** 59K (安全余量)
```

- **关键原则:**

1. **永远留余量** - 不要把上下文填满,留20-30%余量
2. **动态调整** - 根据任务类型调整配比
3. **优先级排序** -
当前任务 > 对话历史 > RAG文档 > 长期记忆

- **记忆Token控制策略:**

| 记忆类型 | 压缩方式 | 目标Token |
|---------|---------|----------|
| 对话历史 | 摘要(旧) + 原文(新3轮) | <15K |
| 工具输出 | 截断(取Top-K) | <5K/次 |
| RAG文档 | 精排(取Top-3) | <30K |
| 长期记忆 | 检索(取相关) | <10K |

- **翻车场景:**
- 工具返回10MB日志 → 撑爆上下文 → 必须截断
- 对话100轮未压缩 → 超限 → 必须摘要
- RAG返回50个文档 → 太多 → 必须重排取Top-5
