# Agent 如何设计记忆机制,避免上下文不断膨胀

- **上下文膨胀问题**:随着对话轮次增加,历史消息累积导致 token 超限、延迟增加、成本上升.

- **分层记忆架构**

1. **工作记忆**
   - 当前对话的最近 N 轮
   - 原始格式保留,精确度高
   - 容量有限(<8K tokens)

2. **短期记忆**
   - 最近几小时/几天的对话摘要
   - 用 LLM 压缩为关键信息点
   - 检索时展开细节

3. **长期记忆**
   - 向量数据库存储历史交互
   - 语义检索召回相关片段
   - 可无限扩展

- **压缩与遗忘策略**

1. **滑动窗口**:保留最近 N 轮,旧的丢弃/压缩
2. **摘要压缩**:每隔 K 轮,LLM 生成摘要替换原始消息
3. **重要性评分**:用 LLM 给消息打分,低分丢弃
4. **实体提取**:维护实体知识图谱('用户叫张三'、'项目是面试题库')
5. **时间衰减**:旧记忆权重降低,模拟人类遗忘

- **实践方案**
```python
class AgentMemory:
    def update(self, message):
        self.working.append(message)
        if len(self.working) > 20:
            old = self.working.pop(0)
            summary = llm.summarize(old)
            self.short_term.append(summary)
            if random() < 0.3:
                embed = llm.embed(summary)
                self.vector_db.insert(embed, summary)
```

- **核心补充：Reflect 机制**
  - **自我反思**：不仅存摘要，还要存“反思”。当 Agent 回答错误或收到负反馈时，触发一个专门的过程，将“错误原因”和“修正策略”写入长期记忆，防止下次再犯。
  - **动态检索**：在构建 Prompt 时，不是简单地回填最近摘要，而是根据当前 User Query 对 Vector DB 进行语义检索，将 Top-K 相关的历史片段召回。这比单纯的时间窗口更有效。
  - **Token 计算与截断**：在 System 层面维护一个 Token 计数器。每次添加 Memory 时预估 Token 量。如果超过上限，优先移除 Working Memory 中最早且“重要性评分”最低的轮次。

- **## 常见考点**
  1. **记忆一致性**：如果用户在较早轮次修改了信息（例如“把我的邮箱改成 B”，后来又改成“C”），检索时召回旧信息（A/B）会导致冲突。如何解决？（在写入时打上时间戳，召回时去重或仅保留最新状态；或者在 KG 层面覆盖实体属性）
  2. **检索相关性**：如何保证检索回来的历史记忆对当前任务是有用的？（使用混合检索：向量检索+关键词过滤，或者引入 Re-ranking 模型对召回结果进行精排）
  3. **跨会话记忆**：用户第二天再来，如何无缝衔接？（必须有 User ID 绑定的长期存储，初始化 Agent 时先加载该用户的 Profile 和近期摘要）
  4. **遗忘策略**：如果 Vector DB 变得非常大，检索速度变慢怎么办？（定期对旧数据进行聚类降维，或设置 TTL 清理无效数据，建立分片索引）
