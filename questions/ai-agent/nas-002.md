# RAGFlow、Dify、FastGPT 等开源 RAG 框架有什么区别?如何选型

- **主流开源 RAG 框架对比**

| 框架 | 定位 | 核心优势 | 适用场景 |
|------|------|----------|----------|
| **RAGFlow** | 深度文档理解 RAG | 基于 DeepDoc 的精准文档解析(表格/图片/布局) | 复杂文档(PDF/合同/论文) |
| **Dify** | LLM 应用开发平台 | 可视化工作流 + Agent + RAG 全栈 | 快速构建 AI 应用 |
| **FastGPT** | 知识库问答 | 工作流编排 + 函数调用 | 企业知识库客服 |
| **LangChain** | LLM 开发框架 | 最灵活的组件化设计 | 定制化 RAG 开发 |
| **LlamaIndex** | 数据框架 | 专注数据连接和索引 | 企业数据湖 RAG |

- **技术架构深度对比**
```text
┌──────────────────────┐  ┌───────────────────────┐  ┌──────────────────────┐
│    LangChain (SDK)   │  │    Dify (Platform)    │  │   RAGFlow (Focus)    │
├──────────────────────┤  ├───────────────────────┤  ├──────────────────────┤
│ • Chains / Runnables │  │ • Visual Workflow     │  │ • DeepDoc (OCR/Layout)│
│ • Integrations (100+)│  │ • RAG Pipeline UI     │  │ • Hybrid Search       │
│ • Memory management  │  │ • Model/Key Mgmt      │  │ • Knowledge Graph     │
└──────────┬───────────┘  └───────────┬───────────┘  └──────────┬───────────┘
           │                          │                         │
           ▼                          ▼                         ▼
    High Code Complexity         Low Code / Ops           High Data Accuracy
```

- **选型决策**

1. **文档复杂度高**(PDF 表格、扫描件)→ **RAGFlow**
- DeepDoc 引擎支持布局识别、表格还原、OCR
- 切片质量显著高于朴素按字数切分

2. **快速构建产品**(工作流 + 多 Agent)→ **Dify**
- 拖拽式工作流设计
- 内置 prompt 模板、模型管理、监控

3. **需要深度定制**(自定义检索逻辑)→ **LangChain**
- 最灵活,但需要更多开发工作
- 适合有工程团队的企业

4. **企业知识库**(客服/FAQ)→ **FastGPT**
- 开箱即用的知识库管理
- 支持函数调用增强

- **## 常见考点**
1. **RAGFlow 的优势原理**：它是如何处理表格错位的？（通过 Vision 模型识别版面，实现基于语义区域的切片，而非单纯依赖文本换行符）
2. **Dify 与 LangChain 的关系**：Dify 底层是否使用了 LangChain？（早期版本可能使用，现在已逐渐解耦或自研编排引擎；重点在于“平台化” vs “库”的区别）
3. **FastGPT 的向量检索策略**：FastGPT 默认使用的重排序机制是什么？（通常混合向量检索与关键词检索，并支持引入 Rerank 模型）
4. **生产环境选型陷阱**：选择 LangChain 直接开发主要面临什么挑战？（版本迭代极快导致代码维护困难，以及抽象层过厚带来的性能损耗）
