# RAG 知识库更新后如何避免旧索引污染

- **索引污染**指知识库更新后,旧版本的向量仍残留在索引中,导致检索到过期或错误信息.

- **污染来源**
1. 文档更新后旧向量未删除
2. 文档删除后向量未清理
3. 版本冲突:新旧版本同时被检索
4. 元数据未同步更新

- **解决方案**
1. **文档级 ID 管理**:每个文档/切片分配唯一 ID,更新时先删旧 ID 再插入新 ID
2. **版本号标注**:向量存储时带 version 字段,检索时过滤旧版本
3. **增量更新管道**:监听文档变更 → diff 检测 → 自动更新索引
4. **TTL 过期机制**:给向量设过期时间,定期清理
5. **软删除 + 定期重建**:标记删除,低峰期全量重建索引
6. **双索引策略**:新旧索引并存,灰度切换后删除旧索引

- **增量更新流程架构**
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                          Document Source (Wiki/S3/DB)                        │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼ (Change Event: Update/Delete)
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Update Orchestrator (Watcher)                        │
│  ┌───────────────────────┐           ┌─────────────────────────────────────┐  │
│  │   Doc Parser & Chunker│           │   Version Manager (Hash/ID Check)   │  │
│  └───────────┬───────────┘           └───────────────┬─────────────────────┘  │
└──────────────┼───────────────────────────────────────┼────────────────────────┘
               │                                       │
               ▼                                       ▼
        ┌─────────────┐                       ┌─────────────────────┐
        │ New Embed   │                       │   Delete Strategy   │
        │ (Generate)  │                       │ (ID / Soft Delete)  │
        └──────┬──────┘                       └──────────┬──────────┘
               │                                         │
               └───────────────┬─────────────────────────┘
                               ▼
        ┌──────────────────────────────────────────────────────┐
│                   Vector Database (Milvus/Qdrant)            │
│  Collection: { id: "doc_v1", vector: [...], is_deleted: 0 }  │
│  Action: UPSERT (Update if exists, Insert if not)            │
└──────────────────────────────────────────────────────────────┘
```

- **工程实践细节**
- **Upsert 语义**：利用 Milvus/Qdrant 的 `upsert` 操作，这比先 `delete` 后 `insert` 性能更高且具备原子性。
- **过滤检索**：在 Metadata 中携带 `doc_id` 和 `timestamp`，查询时虽然不需要显式过滤最新版，但在做特定文档查询时非常有用。
- **Hash 校验**：在解析文档前计算文件 Hash，若 Hash 未变则跳过处理，避免重复 Embedding 消耗 Token。

- **## 常见考点**
1. **性能权衡**：频繁的 Upsert 对向量数据库的写入性能有什么影响？（需要考虑 Segment 合并和索引重建的开销）
2. **最终一致性**：在分布式 RAG 系统中，如何保证文档更新后检索到的即时性？（采用消息队列异步处理 vs 同步处理的选择）
3. **孤儿数据清理**：如果软删除标记失败，如何检测并清理孤立向量？（全量比对或定期脚本扫描）
4. **切片粒度**：如果只更新了文档的一小部分，是否需要重算整个文档的向量？（通常按切片 ID 精度更新，仅重算受影响的 Chunk）
