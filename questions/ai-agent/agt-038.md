# 会话摘要与压缩

摘要与压缩旨在将长对话转化为更短、信息密度更高的表示，以在 Token 预算内保留任务有效信息。

### 核心策略
1.  **LLM 摘要**：周期性或在写入长记前，调用 LLM 提取约束、未决事项和目标。
2.  **增量 vs 全量**：
    *   **增量**：旧摘要 + 新片段 -> 新摘要。省算力，但误差会累积。
    *   **全量**：定期对完整历史重摘要。一致性好，但成本高。
3.  **滚动摘要**：将对话分成若干块，每块生成摘要，再对摘要列表生成总摘要。这是一种折中方案。

### 权衡
摘要越短越省钱，但易丢失细节；越长越保真，但接近上下文上限。工程上常采用「硬约束进结构化字段 + 软偏好进摘要 + 细节进向量库」的分层策略。

### 架构图：增量摘要流程
```text
Original History:
[M1] [M2] [M3] [M4] [M5] [M6] [M7] [M8] [M9] ...
       │
       ├─> (Trigger: Token Limit Reached or N turns)
       │
       ▼
┌───────────────────────────────────────┐
│  Step 1: Select Window (e.g. M1-M5)   │
└───────────────────┬───────────────────┘
                    │
                    ▼
┌───────────────────────────────────────┐
│  Step 2: LLM Summarization            │
│  Input: Current Summary + M1-M5       │
│  Output: New Summary (Condensed)      │
└───────────────────┬───────────────────┘
                    │
                    ▼
New Context:
[New Summary] [M6] [M7] [M8] [M9] ...
```

### 实现细节
*   **触发条件**：可以是固定轮数（如每 10 轮），也可以是 Token 数量接近阈值时。
*   **结构化摘要**：不仅生成文本，还提取 JSON 结构（如 `{"user_goal": "...", "constraints": [...]}`），便于后续程序化处理。

### 代码逻辑（示意）
```python
# 伪代码：增量摘要
# current_summary 可能是 None 或之前的摘要字符串
new_messages = get_recent_messages()

prompt = f"""
Old Summary: {current_summary or '(No history yet)'}
New Conversation:
{format_messages(new_messages)}

Task: Merge the old summary with new conversation into a concise new summary.
Retain critical constraints, user preferences, and pending actions.
"""

new_summary = llm.complete(prompt)
```

### ## 常见考点
1.  **如何避免摘要信息的丢失？**：使用结构化 Prompt 强制 LLM 提取关键字段（如 Intent、Entities）；或者将摘要与重要句子的原句（高亮引用）结合存储。
2.  **摘要的一致性如何保证？**：使用温度较低（T=0.1）的模型进行摘要；定期（如每 50 轮）进行一次全量重算以修正增量误差。
3.  **滚动摘要 vs 增量摘要的区别？**：滚动摘要通常是对多个局部摘要进行再抽象，形成树状结构；增量摘要是线性的“旧摘要+新内容”。树状结构在超长对话下更稳健。
