# Plan-and-Execute 与 ReAct 两种 Agent 模式有什么核心区别？在什么场景下应该选择 Plan-and-Execute？

ReAct 和 Plan-and-Execute 是两种主要的 Agent 任务规划范式。

**ReAct** 是“边想边做”， interleaving 地进行推理和行动。每一步都基于上一步的观测结果决定下一步行动。这种方式灵活、反馈快，适合路径探索或未知环境，但容易陷入局部循环或因早期错误导致后续动作全部失败，Token 消耗不可控。

**Plan-and-Execute** 是“先想后做”，分为两个阶段：
1. **Plan**：模型首先一次性规划出完成任务的全局步骤列表（如：1. 搜索；2. 读取；3. 总结）。这通常利用 LLM 强大的逻辑推理能力生成结构化的任务链。
2. **Execute**：模型按顺序执行步骤，通常不再修改计划（或者引入一个“Re-planner”仅在遇到无法解决的错误时介入）。

**选择场景**：
- **Plan-and-Execute** 适合步骤明确、逻辑链条较长的复杂任务（如编写长篇代码、多步骤的数据分析流程或旅游规划）。它能减少执行过程中的幻觉，避免在中间步骤频繁试错，Token 总成本往往更低且更可控。
- **ReAct** 更适合需要动态调整策略、环境高度不确定的探索性任务（如网页导航、调试未知错误），因为初始计划可能会因为环境变化而完全失效。

## 常见考点
1. **Plan-and-Execute 如果中间某一步失败了怎么办？**
   - 通常做法是让 Executor 将错误反馈给 Planner，触发“重规划”过程，或者在 Plan 阶段就预设条件分支和异常处理逻辑。
2. **为什么 ReAct 容易产生死循环？**
   - 因为模型在 Observation 状态下可能会做出相同的错误推断，陷入 Thought-Action-Observation 的无限循环，通常需要设置“最大步数”限制来兜底。
3. **如何结合两者的优点？**
   - 可以采用“规划-执行-再规划”的混合模式，即有一个大方向的主计划，但在执行每个微步骤时允许使用 ReAct 进行灵活调整。
