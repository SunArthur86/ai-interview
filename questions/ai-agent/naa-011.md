# MCP(Model Context Protocol)的完整架构是什么?它如何解决 N×M 工具集成问题

- **MCP (Model Context Protocol)** 是 Anthropic 提出的开放协议,标准化 LLM 与外部工具/数据的连接方式.

- **N×M 问题**
- N 个 LLM 应用 × M 个工具 = N×M 个集成点
- 每个 LLM 框架(LangChain/OpenAI/GLM)都有自己的工具定义格式
- 工具开发者要为每个框架写适配

- **MCP 架构**

```
LLM 应用 (MCP Client)
↕ MCP 协议 (JSON-RPC)
MCP Server (工具适配器)
↕
实际工具 (DB/API/File/Search)
```

- **核心组件**

1. **MCP Client**:嵌入在 LLM 应用中,负责与 Server 通信
2. **MCP Server**:独立进程,暴露工具能力
3. **Transport**:支持 stdio(本地)和 SSE(远程)
4. **Protocol**:JSON-RPC 2.0

- **MCP 暴露三类能力**

1. **Tools**:可执行的函数(如 query_database)
2. **Resources**:可读取的数据(如文件内容)
3. **Prompts**:预定义的提示模板

- **与 Function Calling 的关系**
- Function Calling:LLM API 层面的工具调用(应用内)
- MCP:跨应用的工具协议(应用间)
- 关系:MCP Server 可以将工具转换为 Function Calling 格式

- **优势**
- 标准化:一次开发,到处可用
- 安全隔离:Server 独立运行,权限可控
- 动态发现:Client 可以列出 Server 提供的工具

- **局限**
- 生态尚早期(主要 Anthropic 生态)
- 协议开销(额外一层 JSON-RPC)
- 多工具场景需要路由层
