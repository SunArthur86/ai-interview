# MCP(Model Context Protocol)的完整架构是什么?它如何解决 N×M 工具集成问题

- **MCP (Model Context Protocol)** 是 Anthropic 提出的开放协议,标准化 LLM 与外部工具/数据的连接方式.

- **N×M 问题**
- N 个 LLM 应用 × M 个工具 = N×M 个集成点
- 每个 LLM 框架都有自己的工具定义格式
- 工具开发者要为每个框架写适配

- **MCP 架构全貌**
```text
┌───────────────────────────────────────────────────────────────────────┐
│                           LLM Application (Host)                       │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │                     MCP Client (e.g. Claude Desktop)             │  │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐   │  │
│  │  │   Local Tool │  │   Resource  │  │      Prompt         │   │  │
│  │  │   Executor   │  │    Reader   │  │    Template         │   │  │
│  │  └──────┬───────┘  └──────┬───────┘  └──────────┬───────────┘   │  │
│  └─────────┼────────────────┼─────────────────────┼───────────────┘  │
└────────────┼────────────────┼─────────────────────┼──────────────────┘
             │                │                     │
             │      JSON-RPC 2.0 (Standardized Protocol)               │
             ▼                ▼                     ▼
      ┌────────────┐   ┌─────────────┐       ┌──────────────┐
      │  stdio     │   │    SSE      │       │   (Other)    │  <-- Transport Layer
      │  (Local)   │   │  (Remote)   │       │              │
      └─────┬──────┘   └──────┬──────┘       └──────────────┘
            │                 │
            ▼                 ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                        MCP Server Ecosystem                             │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────────────┐  │
│  │  File System │  │   Database   │  │   Git / GitHub Integration   │  │
│  │   Server     │  │   Server     │  │         Server               │  │
│  └──────────────┘  └──────────────┘  └──────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────────┘
```

- **核心组件**
1. **MCP Client**:嵌入在 LLM 应用中,负责与 Server 通信
2. **MCP Server**:独立进程,暴露工具能力
3. **Transport**:支持 stdio(本地)和 SSE(远程)
4. **Protocol**:JSON-RPC 2.0

- **MCP 暴露三类能力**
1. **Tools**:可执行的函数(如 query_database)
2. **Resources**:可读取的数据(如文件内容)
3. **Prompts**:预定义的提示模板

- **关键原理细节**
- **Initialization (初始化握手)**: 连接建立后 Client 发送 `initialize` 请求，Server 返回 `capabilities` (支持 tools/resources/prompts 哪些) 和 `serverInfo`。Client 回复 `initialized` 通知。
- **Lifecycle (生命周期管理)**: 支持 `notifications/initialized`, `notifications/progress` 等通知，允许 Server 长运行任务反馈进度。
- **Roots & ListChanges**: 为了资源管理，Client 可以发送 `roots/list` (告知 Server 监听的工作区根目录)，Server 发送 `notifications/list_changed` 通知 Client 数据变更。

- **与 Function Calling 的关系**
- Function Calling:LLM API 层面的工具调用(应用内)
- MCP:跨应用的工具协议(应用间)
- 关系:MCP Server 可以将工具转换为 Function Calling 格式

- **优势**
- 标准化:一次开发,到处可用
- 安全隔离:Server 独立运行,权限可控
- 动态发现:Client 可以列出 Server 提供的工具

- **局限**
- 生态尚早期(主要 Anthropic 生态)
- 协议开销(额外一层 JSON-RPC)
- 多工具场景需要路由层

- **## 常见考点**
1. **传输层差异**：何时使用 `stdio` (本地开发、父子进程通信) 何时使用 `SSE` (云端服务、Web 端集成)？
2. **错误处理**：MCP 如何处理 Server 崩溃或工具执行超时？（通过 JSON-RPC 的 error response 和 Client 端超时机制）
3. **安全性**：LLM 通过 MCP 执行系统命令的风险如何控制？（Server 端权限校验、沙箱运行）
4. **与 OpenAPI 的区别**：MCP 是双向流式交互协议，而 OpenAPI 主要是单向 HTTP 请求描述。
