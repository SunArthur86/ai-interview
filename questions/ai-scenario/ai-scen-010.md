# 如何设计AI Agent的工具调用（Tool Calling）安全边界？防止Agent执行危险操作。

【场景分析】
Tool Calling风险：Agent可能被Prompt注入诱导执行未授权操作、调用错误的工具、传递危险参数。安全边界设计是生产级Agent的核心。

【安全分层架构】
1. 工具注册与权限：
   - 白名单机制：Agent只能调用注册过的工具
   - 分级权限：只读工具（查询）/ 写入工具（创建修改）/ 危险工具（删除/转账）
   - 租户隔离：不同租户可用的工具集合不同
2. 参数校验层：
   - JSON Schema校验：调用参数必须符合预定义Schema
   - 范围限制：金额<上限、时间范围合法、ID格式正确
   - 注入检测：参数值扫描Prompt注入模式
3. 执行前检查：
   - 身份验证：当前用户是否有权限执行该操作
   - 资源归属：操作的资源是否属于当前用户
   - 风险评估：操作的风险等级（low/medium/high/critical）
4. Human-in-the-Loop：
   - 高风险操作：Agent生成执行计划 → 人工确认 → 执行
   - 二次确认：敏感操作（删除、转账）需要额外验证
   - 审计日志：记录完整的调用链（谁、何时、什么参数、结果）

【Prompt注入防御】
- 输入净化：用户输入与系统Prompt明确分隔
- 输出过滤：Agent的Tool Calling参数经过安全过滤器
- 上下文隔离：工具返回结果不直接进入下一轮Prompt（避免间接注入）

【具体实现示例】
```
Agent决策调用delete_file(path)
→ 权限检查：用户是否有delete权限？
→ 参数校验：path是否在允许范围内？
→ 风险评估：delete_file = HIGH RISK
→ 人工确认：发送确认请求给用户
→ 用户确认 → 执行 → 记录审计日志
```

【监控与告警】
- 异常调用模式检测：短时间大量危险操作
- 速率限制：单个用户/会话的工具调用频率
- 熔断：连续失败后暂停Agent执行
