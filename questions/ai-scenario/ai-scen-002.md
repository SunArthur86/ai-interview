# 如何设计一个多模态 RAG 系统？支持图文混合检索，用户可以用文字搜索图片、用图片搜索相关文档。

【场景分析】
多模态RAG挑战：跨模态语义对齐、图文联合索引、多模态上下文组装、模态间引用关联。

【整体架构】
1. 多模态文档摄入：
   - 图片：CLIP/BLIP-2生成图像Embedding + OCR提取文字
   - 表格：结构化解析 → Markdown + 文本摘要
   - 图文混合页：图文分离 + 关联标注
2. 统一向量空间：
   - 使用多模态Embedding模型（CLIP / Jina-CLIP-V2）
   - 文本和图像映射到同一向量空间，支持跨模态检索
   - 向量库：Milvus多向量字段（文本向量 + 图像向量）
3. 多模态检索策略：
   - 文搜文：Text Embedding → 文档chunk
   - 文搜图：Text Embedding → CLIP空间 → 图像
   - 图搜文：Image Embedding → CLIP空间 → 文档
   - 混合搜：图文联合查询 → 多路召回 → 融合排序
4. 多模态生成：
   - 使用多模态LLM（GPT-4o / Claude 3.5 Sonnet）
   - 上下文包含文本chunk + 图像引用 + 表格结构
   - 生成回答时标注图片来源和页码

【关键技术挑战】
- 图文对齐：同一页面的图和文建立关联索引
  - *方案*：在解析阶段生成唯一的 `doc_id` 和 `page_id`，图片和文本块共享此ID，并在元数据中存储相对位置坐标（bbox），以便在检索时重建上下文。
- 分辨率适配：高分辨率图片截取ROI再Embedding
  - *细节*：CLIP通常输入224x224或336x336。对于高清图（如4K PDF），需采用切片策略（如4x4 Grid），生成多个Patch Embeddings，检索时取最大相似度或平均相似度。
- 跨模态排序：图文混合结果的统一相关性排序
  - *细节*：由于向量分数模长可能不一致，需先进行Score Normalization（如Min-Max Scaling），再根据融合权重加权。

【性能优化】
- 图像分块：大图按区域切分，类似文档chunking
- 缓存：CLIP Embedding缓存，避免重复计算
- 近似搜索：HNSW + 量化，百万级图像毫秒检索

## 常见考点
1. **多模态向量模型的选择**：为什么选择CLIP而不是各自用Text和Image模型？（需要强调对齐的Joint Latent Space）。
2. **OCR与视觉特征的结合**：如果图片中包含关键文字（如发票金额），仅靠CLIP向量检索可能不准，如何优化？（答案：OCR结果转文本embedding与Image vector做Co-rerank或加权融合）。
3. **Token限制与上下文压缩**：多模态LLM（如GPT-4o）对图片有Token计费和限制，如何处理大量检索到的图片？（答案：只传递Top-1或Top-2的高清图，其余生成摘要或不传）。
