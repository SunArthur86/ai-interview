# 如何设计AI Agent的记忆系统？支持短期对话记忆和长期知识积累。

【场景分析】
Agent记忆三大需求：短期记忆（当前对话上下文）、工作记忆（任务执行中的中间状态）、长期记忆（跨会话的用户偏好和知识）。

【三层记忆架构】
1. 短期记忆（Working Memory）：
   - 存储：当前对话的消息历史（system + user + assistant）
   - 管理：滑动窗口（保留最近N轮）+ Token预算控制
   - 上下文压缩：超出窗口时用LLM总结早期对话
   - 实现：内存中的消息列表 / Redis
2. 工作记忆（Episodic Memory）：
   - 存储：当前任务的执行轨迹（工具调用、中间结果）
   - 用途：Agent可以回看之前的步骤，避免重复操作
   - 清理：任务完成后归档或删除
   - 实现：任务级别的状态存储
3. 长期记忆（Long-Term Memory）：
   - 类型A - 用户画像：偏好、习惯、历史交互摘要
   - 类型B - 知识记忆：Agent学到的有用信息（如「用户喜欢简洁回答」）
   - 类型C - 事件记忆：重要交互记录（如「上周帮用户做了XX」）
   - 存储：向量化存入向量库 + 结构化存入数据库
   - 检索：当前对话上下文 → 相关记忆检索 → 注入Prompt

【记忆管理策略】
- 写入：不是所有对话都值得记忆 → LLM判断重要性（1-5分）
- 更新：用新信息更新旧记忆（「用户之前住北京」→「现在搬到了上海」）
- 遗忘：低频访问 + 过期记忆 → 软删除或归档
- 隐私：敏感信息加密存储，支持用户查看和删除

【实现方案】
- MemGPT模式：操作系统式的内存管理（分页、缓存、写入磁盘）
- LangChain Memory：ConversationBufferMemory / ConversationSummaryMemory
- 自研：Redis（短期）+ 向量库（长期）+ PostgreSQL（结构化）

【关键权衡】
- 记忆越多 → 上下文越长 → 成本越高、延迟越大
- 记忆太少 → Agent「健忘」→ 用户体验差
- 平衡点：每轮注入3-5条最相关的长期记忆
