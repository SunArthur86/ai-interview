# 如何评估微调效果？有哪些关键指标和注意事项？

🎯 **本质**：LLM微调评估需要多维度衡量——不能只看loss，要全面评估能力保持、领域提升和潜在退化。

📊 **评估维度**：

**1. 自动评估**
  领域任务：
  - 选择该领域的标准基准（如医学: MedQA, 法律: LawBench）
  - 与基座模型对比，确认微调有效
  
  通用能力保持：
  - MMLU（多任务理解）
  - GSM8K（数学推理）
  - HumanEval（代码生成）
  - 确保微调没有"灾难性遗忘"
  
  安全评估：
  - 对齐性（是否产生有害内容）
  - 鲁棒性（对抗prompt测试）

**2. 人工评估**
  盲评（A/B Test）：
  - 同时展示基座模型和微调模型的回答
  - 标注者选择更好的一个（Elo rating）
  - 金标准但昂贵
  
  维度评分：
  - 准确性（事实是否正确）
  - 完整性（回答是否全面）
  - 流畅性（语言是否自然）
  - 相关性（是否切题）

**3. LLM-as-Judge**
  用GPT-4等强模型评估微调模型输出
  方法：
  - pairwise comparison（两两比较）
  - pointwise scoring（单独打分）
  - reference-based（与标准答案对比）
  注意：Judge模型不应与被评估模型相同

**4. 训练指标**
  Loss曲线：
  - Train loss持续下降 -> 正常
  - Val loss上升 -> 过拟合，停止训练
  
  评估频率：
  - 每N步评估一次
  - 保存最佳checkpoint（而非最后一个）

**常见问题诊断**：
| 问题 | 症状 | 原因 | 解决 |
|------|------|------|------|
| 过拟合 | Train loss降但效果差 | 数据太少/epochs太多 | 减少epochs/增加数据 |
| 灾难性遗忘 | 通用能力下降 | 只用领域数据 | 混入通用数据 |
| 格式不稳定 | 输出格式不一致 | 数据格式不统一 | 统一数据格式 |
| 模式坍缩 | 总是输出相似内容 | 数据多样性不足 | 增加数据多样性 |
| 不收敛 | Loss不降 | 学习率不对 | 调整学习率 |

**最佳实践**：
1. 建立评估流水线：微调前测基线 -> 微调后对比
2. 保留Hold-out测试集（不在训练中的评估数据）
3. 持续监控：上线后持续收集bad case
4. 版本管理：记录每次微调的数据、超参和效果
