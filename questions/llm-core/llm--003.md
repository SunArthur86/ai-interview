# 大模型参数量为何常设计为 7B、13B、33B、65B 等数值?这些数字背后有什么讲究

- **参数规模设计的工程考量**

1. **GPU 显存对齐**
   - 单张 A100/H100 = 40GB/80GB
   - FP16 下: 7B≈14GB、13B≈26GB、70B≈140GB
   - 7B/13B 可单卡运行，70B 需 2-4 卡 TP
   - 设计目标: 参数量 × 2 bytes ≈ N × 单卡显存

2. **训练效率 (Chinchilla Laws)**
   - DeepMind 研究表明: 最优训练是参数量与数据量按 ~20:1 的 token:param 比例
   - 7B 模型 → ~140B tokens
   - 70B 模型 → ~1.4T tokens

3. **推理成本**
   - 推理成本 ∝ 参数量
   - 7B 适合大规模部署(边缘设备/低成本)
   - 70B 适合高质量场景

4. **DeepSeek 的创新**
   - MoE 架构打破线性关系: 总参数大但激活参数小
   - DeepSeek-V3: 671B 总参数但仅 37B 激活
   - 推理成本接近 37B 稠密模型

5. **实际约束**
   - 并行度: 参数量需要能被 TP/PP 度数整除
   - 带宽: 跨卡通信量与参数分布相关

- **趋势**: MoE 让'总参数'和'推理成本'解耦，未来规模设计会更灵活。
