# 滑动窗口（Sliding Window）注意力机制（如 Longformer, Mistral）是如何将计算复杂度从 O(N^2) 降低到 O(N) 的？它对模型的上游预训练有什么特殊要求？

滑动窗口注意力机制将 Self-Attention 的计算限制在每个 Token 周围的一个固定大小窗口（如 4096）内，而非计算所有 Token 两两之间的注意力。这样每个层只计算局部注意力，复杂度随序列长度线性增长。为了保留全局信息，这些模型通常每隔一层增加一个全局注意力 Token 或 Dilated Sliding Window（空洞卷积式的滑动窗口）来扩大感受野。对上游预训练的特殊要求是：模型在预训练时必须已经使用了这种稀疏注意力模式，或者通过适配训练适应这种模式。因为直接将原本全量注意力训练的模型权重迁移到滑动窗口结构上，会导致模型无法捕捉长距离依赖，性能剧烈下降。

**实战案例**：在尝试将标准的 Llama-2-7B 推理改造为 Mistral 风格的滑动窗口机制时，我们直接修改了 Attention Mask 限制为 4096 窗口。结果模型在处理长代码文件时完全丢失了上下文，因为 Llama-2 的 Head 权重在全量 Attention 下学习到了全局依赖，强行稀疏化导致权重与 Mask 不匹配，必须进行 SFT（监督微调）才能恢复效果。

**代码示例**：
```python
# FlashAttention 中的滑动窗口 Mask 生成逻辑
def get_sliding_window_mask(seq_len, window_size):
    # 创建下三角矩阵 (Seq, Seq)
    mask = torch.tril(torch.ones(seq_len, seq_len))
    # 将窗口外的位置置为 -inf
    for i in range(seq_len):
        start = max(0, i - window_size + 1)
        mask[i, :start] = float('-inf')
    return mask
```

**对比表格**：

| 特性 | 全局注意力 | 滑动窗口注意力 |
| :--- | :--- | :--- |
| **计算复杂度** | O(N²) | O(N × W) (W为窗口大小，近似 O(N)) |
| **显存占用** | 极高，随长度平方增长 | 较低，随长度线性增长 |
| **长距离依赖** | 天然支持，效果最好 | 依赖堆叠层或全局 Token 增强感受野 |
| **预训练要求** | 标准 Transformer 预训练 | 必须在预训练阶段引入稀疏 Mask |
| **推理吞吐量** | 受限于 KV Cache 的大小 | KV Cache 可以截断，吞吐量极高 |
