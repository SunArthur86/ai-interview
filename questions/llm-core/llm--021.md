# 什么是上下文窗口(Context Window)?为什么它有限制

- **上下文窗口** 是大模型一次能处理的最大 Token 数量.

- **为什么有限制?**
  - **Self-Attention 计算复杂度是 O(n²)**:
    - 在 Transformer 的每一层中，每个 Token 都需要与序列中所有其他 Token 计算注意力得分（QK^T）。
    - 序列长度翻倍，计算量会变为原来的 4 倍，导致推理延迟显著增加。
  - **KV Cache 随序列增长占用显存**:
    - 为了加速推理，系统会缓存 Key 和 Value 向量。
    - 显存占用 ≈ 2 × 层数 × 隐藏层维度 × 序列长度 × 精度字节。
    - 序列长度直接决定了显存瓶颈，限制了 Batch Size 的大小。
  - **序列越长,注意力越分散**:
    - "Lost in the Middle" 现象：模型倾向于关注开头和结尾的信息，中间的关键信息容易被忽略，导致长文本理解能力下降。

- **常见窗口大小**
  - GPT-3: 4K tokens
  - GPT-4 Turbo: 128K tokens
  - Claude 3: 200K tokens
  - Gemini 1.5: 1M~2M tokens

- **窗口不够怎么办?**
  - **RAG**:只检索相关片段，减少无效 Token 输入。
  - **滑动窗口**:分段处理，保持模型关注局部上下文（如 Longformer, Sliding Window Attention）。
  - **上下文压缩**:摘要+关键信息，将长文档压缩为高密度向量或摘要。

- **一句话理解**:上下文窗口 = 大模型的'工作记忆容量',类似人一次能记住多少页书.

## 常见考点
1.  **KV Cache 的具体作用是什么？** （解释其如何避免重复计算 Key 和 Value，以及它带来的显存压力）。
2.  **Ring Attention 是如何突破显存限制的？** （了解分布式长上下文训练方案，将序列切块分到不同 GPU 上计算 Attention）。
3.  **什么是 FlashAttention？** （了解其通过 Tiling 技术优化 Attention 计算，减少 HBM 访问次数，从而加速计算并降低显存占用）。
4.  **RoPE (旋转位置编码) 如何处理长文本外推？** （线性内插与 NTK-Aware Scaled RoPE 等动态扩展长度的技术）。
