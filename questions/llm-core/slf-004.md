# 什么是长上下文（Long Context）技术？

长上下文（Long Context）指LLM能处理的输入token数从4K扩展到128K-1M+：

**技术挑战：**
1.  **计算复杂度**：标准Attention是O(n²)，序列长度翻倍，计算量翻四倍。
2.  **KV Cache内存**：显存占用随序列长度和Batch Size线性增长，主要瓶颈在于推理时的KV Cache存储。
3.  **训练数据**：缺乏长文本训练数据，且长距离依赖难以学习（大海捞针 Needle-in-a-Haystack 挑战）。
4.  **召回率下降**：随着Context变长，模型对中间信息的检索能力往往减弱（Lost in the Middle现象）。

**优化技术：**
1.  **位置编码扩展**：
    *   **RoPE外推**：NTK-aware scaling（通过修改基频扩展位置编码范围）、YaRN（在Attention logits上插值）。
    *   **ALiBi**：通过线性偏置替代绝对位置编码，具备天然外推能力。
2.  **稀疏注意力**：
    *   **Sliding Window（Mistral）**：每个Token只关注相邻窗口内的Token。
    *   **StreamingLLM**：保留Attention Sink（起始Token）和滑动窗口内的Token，丢弃中间Token以保持缓存稳定。
3.  **Ring Attention**：
    *   将序列切分到不同GPU，设备间只传递KV Cache块，环形通信，实现跨GPU的线性扩展注意力计算。
4.  **KV Cache压缩**：
    *   **H2O (Heavy Hitter Oracle)**：保留高频关注的Token（Heavy Hitters）和最近的Token，丢弃其他Token。
    *   **StreamingLLM**：同样是基于Attention Sink的保留策略。
5.  **FlashAttention**：优化内存访问，将Attention计算从O(n²)显存降低到O(n)，并利用GPU Triton进行加速，是长上下文训练和推理的基石。

**模型对比：**
- GPT-4 Turbo：128K
- Claude 3：200K
- Gemini 1.5 Pro：2M tokens (1M Audio/Video)
- Llama 3.1：128K
- Qwen2.5：128K
- GLM-4-Long：1M

**应用场景：**
- 长文档分析（作为RAG替代方案，减少检索切分带来的语义丢失）
- 代码仓库理解（全库上下文）
- 多轮对话（不需要摘要，保留历史完整信息）
- Few-shot示例池扩大（提供更多In-context Learning示例）

**趋势：**Native Long Context（原生长上下文）比RAG更简单，但成本更高；混合架构（长Context + RAG）是未来的主流方向。
