# 什么是Prefill-Decode分离?为什么分离

Prefill-Decode分离是将大模型推理分为“预填充”（处理输入Prompt）和“解码”（逐个生成输出Token）两个阶段。Prefill是计算密集型任务，主要负责并行处理输入并生成KV Cache；Decode是访存密集型任务，负责基于KV Cache逐Token生成文本。分离两者是为了针对各自不同的资源瓶颈（算力vs带宽）进行独立的优化和调度，从而提升整体效率和吞吐量。

**一、两个阶段的深度解析**

**1. Prefill 阶段（预填充）**
*   **任务：** 处理用户输入的 Prompt。Prompt 中的所有 Token 是并行输入模型的。
*   **计算模式：** **计算密集型**。因为输入长度通常较长（如几K tokens），Attention 计算涉及大矩阵乘法（$Q \times K^T$），能够充分利用 GPU 的 Tensor Core 进行高性能并行计算。
*   **产出：** 生成 KV Cache（Key 和 Value 的缓存矩阵），供后续 Decode 阶段复用。

**2. Decode 阶段（解码）**
*   **任务：** 自回归地逐个生成 Token。每次生成一个新 Token。
*   **计算模式：** **访存密集型**。
    *   每一步只输入 1 个 Token，计算量极小（矩阵乘法维度很小）。
    *   但为了计算 Attention，必须将当前这一个 Token 的 Q 与之前所有 Token 的 K、V 进行运算。这意味着每一步都要从显存（HBM）中读取巨大的 KV Cache（例如已生成了 1000 个 token，就要读回 1000 组 KV 向量）。
    *   **瓶颈：** GPU 算力往往处于闲置状态，等待显存数据传输，即“内存墙”问题。

**三、为什么要分离？**

1.  **瓶颈不同，优化策略不同：**
    *   **Prefill 优化：** 侧重于算力优化。如使用 FlashAttention 减少显存访问、PagedAttention 等技术加速矩阵运算。
    *   **Decode 优化：** 侧重于显存带宽优化。如 KV Cache 量化（FP16 -> INT8）、投机采样、连续批处理。

2.  **调度与并发：
    *   在 LLM 服务框架（如 vLLM, TGI）中，分离可以实现 **Continuous Batching（连续批处理）**。
    *   可以在一个 Batch 中混合处理：有的请求正在做 Prefill（并行计算），有的请求正在做 Decode（串行访存），最大化 GPU 利用率。
    *   如果不分离，简单串行处理会导致当某个请求处理长 Prompt 时，后续所有生成中的请求都被阻塞（首尾延迟高）。

3.  **资源利用率：** Prefill 阶段通常很快结束，Decode 阶段耗时极长（生成 100 个 token 需要 100 步）。分离管理可以防止 Decode 阶段的低算力利用率拖累整体系统的吞吐量评价。

## 常见考点
1.  **Time to First Token (TTFT)：** 指的是从发送请求到收到第一个输出 Token 的时间。这个时间主要由哪个阶段决定？（主要由 Prefill 阶段决定，受 Prompt 长度影响大）。
2.  **Inter-token Latency：** 指的是生成后续每个 Token 的延迟。由什么决定？（主要由 Decode 阶段的显存带宽和 KV Cache 大小决定）。
3.  **显存占用的主要来源：** 在推理阶段，显存主要被 KV Cache 占用还是模型权重占用？（对于长文本生成，KV Cache 往往比模型权重本身占用更多显存，因此需要 PagedAttention 等管理技术）。
