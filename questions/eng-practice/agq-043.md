# 模型推理速度是怎么优化的

**Situation：** 本地部署的模型(用于 embedding、reranking 等)推理速度需要优化.
**Task：** 提升本地模型的推理速度,降低延迟.
**Action：** 
1. 模型量化:
  *   Embedding 模型(BGE-M3)使用 INT8 量化.
  *   **细节**：将模型权重从 FP32/FP16 转为 INT8，利用 Tensor Core 加速 INT8 矩阵运算。校准集用于确定量化缩放因子，确保精度损失 < 0.5%,推理速度提升 40%.

2. 推理引擎优化:
  *   使用 ONNX Runtime 替代 PyTorch 推理(去除了 Python GIL 锁和训练相关开销,速度提升 30%).
  *   **加速后端**：启用 CUDA Graphs 减少 Kernel 启动开销，对稳定形状输入使用 TensorRT 进行进一步优化.

3. 批处理:
  *   **静态批处理**：Embedding 计算支持 `batch_size=64`; Reranking 支持批量推理(一次性对 40 个候选打分).
  *   **动态批处理**：设置 `max_batch_size` 和 `timeout_ms`。推理引擎在超时前积累请求，凑齐一批处理，最大化 GPU 利用率.

4. 模型预加载:
  *   服务启动时预加载模型到 GPU 显存,避免首次请求的冷启动延迟(解决 Page Fault 导致的卡顿).

**推理流程架构图：**
```text
Requests Stream
    │
    ▼
[Dynamic Batching Queue]
    │ (Accumulate or Timeout)
    ▼
[Preprocessing] (Tokenization)
    │
    ▼
[ONNX Runtime / TensorRT]
    │ (INT8 Quantized Weights)
    │
    ▼
[Postprocessing] (Normalization)
    │
    ▼
Response
```

**Result：** 
*   Embedding 计算延迟:单条 15ms → 批量 1000 条 3s(均摊 3ms/条).
*   Reranking 延迟:40 条候选 120ms(batch 推理).
*   GPU 利用率从 30% 提升到 75%.

## 常见考点
1.  **KV Cache**：在生成式 LLM 中，KV Cache 是什么？如何优化显存占用（如 PagedAttention）？
2.  **Continuous Batching**：在 LLM 推理中，为什么 Continuous Batching（连续批处理）比 Static Batching 效率更高？（解决“长短不一”的 Pad 问题）。
3.  **AIGC vs 非生成式**：Embedding 和 Rerank 模型的优化与 Chat 模型的主要区别（无 Decode 阶段，主要瓶颈在计算）。
