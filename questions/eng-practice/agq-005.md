# 意图识别模块是怎么实现的

**Situation：** 企业客服场景中,用户意图多样(咨询、投诉、查询订单、操作请求等),且存在大量隐含意图和多意图混合的情况(如"我的订单怎么还没到,我要退款"同时包含查询和操作意图).

**Task：** 设计一个准确率高、可扩展的意图识别模块,能处理多意图和隐含意图.

**Action：** 
1. 两阶段意图识别架构:
第一阶段 -- 粗分类(基于规则 + 小模型):
正则匹配高频模式(如订单号格式 [A-Z]{2}-\d{8}-\d{3} ).
FastText 轻量分类器做初步意图分类(延迟 < 5ms).
覆盖 80% 的常见查询,快速响应.

第二阶段 -- 精细分类(基于 LLM):
对第一阶段无法确定的复杂查询,调用 LLM 进行意图分析.
使用结构化 Prompt,输出标准 JSON:{"intents": [{"type": "...", "confidence": 0.9, "entities": {...}}]}
支持多意图识别和实体提取.

2. 意图体系设计(三级分类):
**L1：** 咨询/操作/投诉/闲聊
**L2：** 产品咨询/技术咨询/订单查询/退款申请/...
**L3：** 具体产品线/具体操作类型/...

3. 置信度阈值策略:
confidence ≥ 0.85 → 直接执行.
0.6 ≤ confidence < 0.85 → 向用户确认意图.
confidence < 0.6 → 追问澄清.

**Result：** 
意图识别准确率达到 94%(测试集 5000 条标注数据).
多意图识别 F1-score 达到 0.89.
第一阶段处理了 80% 查询,LLM 调用量减少 80%,节省了大量 token 成本.

**原理细节补充：**
*   **FastText 参数调优**：使用 `wordNgrams=2` 捕捉局部语序特征，针对中文采用 `jieba` 分词后输入，`lr=1.0`, `epoch=25` 以保证在短文本上的收敛速度和准确率。
*   **LLM Prompt 优化**：采用 `CoT (Chain of Thought)` 引导模型先分析关键词，再判断意图，最后提取实体，减少幻觉。例如："请先分析用户是否表达了负面情绪，再判断属于投诉还是咨询"。
*   **Bad Case 处理**：针对"卡顿"、"打不开"等歧义词，在粗分类阶段增加领域词典映射，确保优先路由到技术支持大类。

**## 常见考点**
1.  **长尾意图处理**：当遇到训练集中未见过的新意图时，系统如何降级处理？（答：置信度低触发澄清，并自动加入待标注队列用于模型迭代）。
2.  **多意图冲突解决**：如果用户同时说"查账单"和"退款"，优先级怎么定？（答：通常根据业务重要性定义优先级，如退款 > 查询，或者根据意图在Prompt中出现的顺序或置信度加权排序）。
3.  **实时性优化**：除了两阶段架构，还有哪些手段降低延迟？（答：Prompt缓存、量化模型、VLLM推理加速）。
4.  **数据回流机制**：如何利用线上修正的数据？（答：构建"人机协同"标注平台，将置信度低且人工修正过的Query自动写入训练集，定期Fine-tune小模型）。
