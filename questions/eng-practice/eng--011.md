# 请设计一个企业级智能客服 Agent 平台的完整架构,支持多租户和水平扩展.

- **架构设计**

  - **接入层**
    - API Gateway: 认证、限流、路由
    - WebSocket: 支持流式输出

  - **核心层**
    - **意图识别**: BERT 分类器(8类意图) → 路由到对应处理链
    - **Agent 编排器**(状态机 + DAG 混合)
    - ReAct Agent: 通用问答
    - RAG Agent: 知识库检索
    - SQL Agent: 数据查询
    - 反思 Agent: 质量校验

  - **服务层**
    - **多路检索引擎**: 向量 + BM25 + 混合 RRF
    - **记忆系统**: Redis(短期) + Milvus(长期)
    - **工具系统**: 搜索/计算器/DB 查询 + MCP
    - **模型路由**: 多模型候选 + 三态熔断器

  - **支撑层**
    - **文档 ETL**: PDF/Word → 分块 → 向量化 → 入库
    - **全链路追踪**: OpenTelemetry
    - **配置中心**: Prompt/工具/模型版本管理

- **多租户方案**
  - 数据隔离: 每租户独立 Collection(Milvus)
  - Prompt 隔离: 每户可自定义 system prompt
  - 权限隔离: RBAC + API Key
  - 资源配额: 每租户 QPS/Token 限制

- **水平扩展**
  - 无状态 API(状态存 Redis)
  - Milvus 集群分片
  - 模型推理用 vLLM + Tensor Parallel

- **关键指标**
  - P95 延迟 < 3s
  - 幻觉率 < 5%
  - 可用性 99.9%
  - 支持 1000+ 并发

  ## 常见考点
  1.  **大模型推理的高并发如何处理？**
      - 使用 vLLM 或 TGI 等高性能推理框架；启用 Continuous Batching 机制以提高 GPU 利用率；对于极高并发，可采用 Tensor Parallelism 或 Pipeline Parallelism 进行模型分片。
  2.  **多租户数据隔离的粒度如何选择？**
      - 高安全需求（如金融）：物理隔离或独立 Database/Collection；一般需求：逻辑隔离（TenantId 字段）；向量库通常建议独立 Collection 以避免索引污染和数据混淆。
  3.  **如何保证长对话的上下文管理？**
      - 使用 Redis 存储短期会话摘要和历史消息；对于超长历史，采用滑动窗口或摘要机制，将早期对话压缩为 Summary 传入 LLM。
