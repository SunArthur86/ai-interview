# 数据丢失和恢复策略是什么

**Situation：** 关键数据(对话历史、知识库索引、用户偏好等)的丢失会严重影响服务质量.

**Task：** 建立完善的数据备份和恢复机制.

**Action：** 
1. **数据分级:**
   - **核心数据(RPO=0):** 知识库原始文档、用户配置 → 实时同步备份.
   - **重要数据(RPO<1h):** 对话历史、向量索引 → 每小时增量备份.
   - **可重建数据(RPO<24h):** 缓存数据、统计报表 → 每日备份.

2. **备份策略:**
   - **PostgreSQL：** 流式复制(主从同步)+ 每日全量备份 + WAL 归档.
   - **Milvus：** 定期 snapshot + 原始文档可重建索引.
   - **Redis：** RDB 快照(每小时)+ AOF 持久化.

3. **降级策略(补充说明):**
   - L0(正常): 所有功能正常运行
   - L1(轻微): 关闭非核心功能(如推荐、个性化)
   - L2(中度): 模型降级(复杂查询也走快速模型)
   - L3(严重): 只返回缓存和FAQ回答,不调用LLM

4. **恢复演练:**
   - 每季度做一次灾难恢复演练.
   - **模拟场景：** 数据库主节点故障、向量库数据损坏、Redis 全部清空.
   - 记录恢复时间(RTO),确保满足 SLA 要求.

**Result：** 
- 核心数据零丢失(RPO=0,主从同步).
- 数据库故障恢复时间 < 5 分钟(主从切换).
- **向量库重建时间：** 约 6 小时(从原始文档重新计算 embedding).

## 常见考点
1. **RPO 与 RTO 的区别**：请解释两者的含义及在你的系统中的具体数值。RPO (Recovery Point Objective) 指恢复点目标（业务能容忍的数据丢失量，如 0 或 1小时）；RTO (Recovery Time Objective) 指恢复时间目标（服务恢复所需时长，如 5分钟）。
2. **备份文件校验**：备份文件损坏了怎么办？必须定期（如每周）对备份文件进行完整性校验（如 `pg_verifybackup` 或 checksum 验证），确保备份是可用的。
3. **跨地域容灾**：如果整个机房发生故障（如断电、火灾），你的备份数据是否安全？需要提到“异地备份”或“冷备/热备方案”，即关键数据需异步复制到不同 Region（如从华东同步到华南）。
