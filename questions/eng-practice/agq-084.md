# KV Cache 是什么?为什么需要

**1. 问题背景：**
Transformer 的 Self-Attention 计算中，每个新 token 需要和之前所有 token 计算注意力。
自回归生成时，第 $t$ 个 token 的 $K、V$ 计算依赖于前 $t-1$ 个 token。如果不缓存，每生成一个 token，都需要重新计算所有历史 token 的 $K, V$ 矩阵。

**2. KV Cache 原理：**
在自回归生成过程中，缓存每一层的 $K$ (Key) 和 $V$ (Value) 矩阵。
生成第 $t+1$ 个 token 时：
- 只需要计算新 token 的 $Q_{new}、K_{new}、V_{new}$。
- 将 $K_{new}、V_{new}$ 与 缓存的 $[K_{1:t}, V_{1:t}]$ 拼接。
- 使用完整的 $K$ 和 $Q_{new}$ 计算注意力得分。
- 将新的 $K、V$ 追加到缓存中。
避免了重复计算前 $t$ 个 token 的 $K$ 和 $V$ 的矩阵乘法运算。

**3. 计算复杂度对比：**
- 不使用 KV Cache：生成 $n$ 个 token 的总计算量 $\propto \sum_{i=1}^n i^2 = O(n^3)$ (实际中 Attention 部分是 $O(n^2)$，但重复计算导致常数项巨大)。
- 使用 KV Cache：每个新生成的 token 只需计算当前 token 的 Attention，总计算量 $\propto O(n^2)$。

```text
无 Cache:
Step 1: Token1 -> [Attn(1)]
Step 2: Token1+Token2 -> [Attn(1,2)]  (重复计算 1 的 K,V)
Step 3: Token1+Token2+Token3 -> [Attn(1,2,3)] (重复计算 1,2 的 K,V)

有 KV Cache:
Step 1: Token1 -> [Attn(1)] -> Cache(K1, V1)
Step 2: Token2 -> [Attn(Cache(K1)+K2)] -> Cache(K1, K2)
Step 3: Token3 -> [Attn(Cache(K1,K2)+K3)] -> Cache(K1, K2, K3)
```

**4. 内存开销 (显存瓶颈)：**
KV Cache 大小 $= 2 \times \text{num\_layers} \times \text{num\_heads} \times \text{seq\_len} \times \text{head\_dim} \times \text{batch\_size} \times \text{dtype\_size}$。
例如 GPT-3 (175B, 96 layers, 128 heads, 128 dim, FP16) 在 2K 长度下，单 batch 的 KV Cache 约为：
$2 \times 96 \times 128 \times 2048 \times 128 \times 2 \text{ bytes} \approx 150 \text{ GB}$ (未优化)。
这是大模型推理需要大量显存的原因之一（往往超过模型权重本身）。

**5. 优化技术：**
- **MQA (Multi-Query Attention) / GQA (Grouped Query Attention)：**
  让多个 Query Head 共享同一组 Key 和 Value。例如 GQA 将 $K, V$ 的 Head 数量减少，大幅降低 Cache 大小，且对性能影响很小。
- **PagedAttention (vLLM)：**
  借鉴操作系统虚拟内存的思想，将 KV Cache 分页管理，解决显存碎片问题，提高显存利用率，实现高效的连续批处理。
- **KV Cache 量化 (FP8 / INT8)：**
  将 FP16 的 KV Cache 量化为 INT8 或 FP8，减少一半以上的显存占用。可能会引入微小的精度损失。
- **FlashAttention：**
  虽然 FlashAttention 主要优化计算速度（减少 HBM 访问），但也支持与 KV Cache 结合，优化 Attention Kernel 的实现。

**## 常见考点**
1. **KV Cache 的内存增长**：KV Cache 占用的显存与什么成正比？
   - *答案*：与序列长度（seq_len）、batch size 和层数成正比。因此长文本推理对显存压力极大。
2. **Prefill 阶段 vs Decode 阶段**：KV Cache 在这两个阶段的作用有何不同？
   - *答案*：在 Prefill 阶段（处理 Prompt），需要计算所有输入 token 的 KV 并写入 Cache，计算密集型；在 Decode 阶段（生成），每个 token 读取 Cache 并计算，访存密集型（Memory-bound）。
3. **MQA/GQA 对显存的影响**：为什么能减少显存？
   - *答案*：因为 $K, V$ 的存储维度减小了。原本 $N$ 个 head 存 $N$ 份 KV，现在可能只存 1 份或 $N/k$ 份，显存占用直接除以分组数。
