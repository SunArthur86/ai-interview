# 如何做的性能基准测试

**Situation：** 需要建立性能基准,作为优化的参照和回归检测的标尺.
**Task：** 建立可重复的性能基准测试体系.
**Action：** 
1. **基准测试场景:**
**单组件基准：** 向量检索延迟（P99）、LLM 调用延迟（TTFT/TBT）、Reranking 延迟.
**端到端基准：** 简单问答延迟、RAG 问答延迟、工具调用延迟.
**压力基准：** 最大 QPS、最大并发数、错误率阈值.
2. **基准测试工具:**
自研基准测试脚本(Python pytest-benchmark)。
**压力测试：** Locust 模拟用户行为。
**结果存储和对比：** 自建 Dashboard 或 Prometheus + Grafana。
3. **基准测试执行:**
每次重大变更前后各跑一次（集成在 CI pipeline 中）。
结果存入数据库,支持历史对比。
**设置性能回归阈值：** 延迟增加 > 10% 或吞吐量下降 > 5% 触发告警，阻断合并。
4. **基准数据集:**
固定的 100 条测试 query(覆盖各种类型，长短不一)。
每次基准测试使用相同的数据集,确保可比性；Mock 外部依赖（如 LLM API）以消除网络波动影响。

**基准测试流程图：**
```text
┌─────────────┐    ┌─────────────┐    ┌──────────────┐
│ Code Commit │───>│ Benchmarks  │───>│  Report &    │
└─────────────┘    │ (Locust/    │    │  Compare     │
                   │  pytest)    │    └──────┬───────┘
                   └──────┬──────┘           │
                          │                 ▼
                   ┌──────▼──────┐    ┌──────────┐
                   │ Mocked Data │───>│ Pass/Fail│
                   └─────────────┘    └──────────┘
```

**Result：** 
建立了完整的性能基线数据。
检测出了 2 次因代码变更导致的性能回归。
性能优化有了量化的参照标准。

## 常见考点
1. **如何消除外部环境对基准测试的干扰？**
   *追问：测试环境是否与生产环境同构？如何 Mock LLM 的响应时间？
2. **如何选择性能指标？**
   *追问：为什么关注 P99 而不是平均值？TPS、QPS 和 Latency 的关系是什么？
3. **冷启动对测试结果的影响？**
   *追问：如何处理 JIT 编译、连接池预热等问题？是否有“预热”阶段？*
