# 数据库查询是怎么优化的

**Situation：** 随着数据量增长,部分数据库查询变慢,影响 Agent 工具调用的响应速度.
**Task：** 优化数据库查询性能,确保工具调用延迟在可接受范围内.
**Action：** 
1. 向量数据库优化:
  *   **索引选择：** 数据量 < 100 万用 HNSW(延迟低),> 100 万用 IVF_SQ8(内存占用小).
      *   *原理细节*：HNSW 基于分层图结构，查询时间复杂度约为 O(log N)；IVF_SQ8 基于倒排文件和标量量化，通过牺牲极小精度换取内存大幅降低（约 1/4）。
  *   **HNSW 参数调优：** `M=16`(每节点连接数，影响召回率与构建时间), `efConstruction=200`(构建时的搜索宽度), `ef=128`(查询时的搜索宽度，越高越准但越慢).
  *   **隔离策略：** Collection 按租户分,利用 PartitionKey 隔离数据,避免跨租户查询扫描.

2. 关系型数据库优化:
  *   **索引策略：** 为高频查询添加组合索引,遵循“最左前缀”原则; 针对范围查询优化索引顺序.
  *   **分析工具：** 使用 `EXPLAIN ANALYZE` 分析慢查询,重点关注 `Seq Scan`(全表扫描)和 `Heap Fetches`(脏页读取).
  *   **分区表：** 对话历史表按月分区(partitioning),大幅减少查询扫描的数据量(时间条件过滤).
  *   **连接池配置：** `min=5`, `max=20`, `max_idle=300s`. 防止连接频繁建立带来的 TCP/TLS 握手开销.

3. Redis 优化:
  *   **数据结构：** 缓存用 String,会话状态用 Hash(减少 key 数量),排行用 Sorted Set.
  *   **Key 设计：** 采用层级规范 `{service}:{tenant}:{type}:{id}`,便于管理。
  *   **高并发防护：** 引入本地 Caffeine 缓存 + 热点 Key 探测,防止 Big Key 阻塞 Redis 主线程.

4. 查询模式优化:
  *   **防 N+1：** 批量查询替代循环查询,将多次 `SELECT` 合并为 `IN (...)` 查询.
  *   **预编译：** 使用 PreparedStatement,利用计划缓存减少 SQL 解析开销.
  *   **列裁剪：** 严禁 `SELECT *`,只查询业务需要的字段,减少网络传输和内存开销.

**架构数据流简图：**
```text
Client Request
    │
    ▼
[App Layer] (Check L1 Cache: Caffeine)
    │ Miss
    ▼
[Redis Layer] (Check L2 Cache: String/Hash)
    │ Miss
    ▼
[DB Layer]
    ├─> [PG] (Partitions + Indexes + Connection Pool)
    └─> [Milvus] (Tenant Partition -> HNSW Index -> IVF Search)
    │
    ▼
Response
```

**Result：** 
*   向量检索 P99 延迟稳定在 50ms 以内.
*   SQL 慢查询数量从日均 200 降到 5.
*   Redis 缓存命中率 95%,读延迟 < 1ms.

## 常见考点
1.  **索引失效场景**：为什么有时候加了索引还是慢？（如使用函数计算、隐式类型转换、like '%xxx'）。
2.  **Redis 穿透/击穿/雪崩**：缓存失效时的防护措施是什么？（布隆过滤器、互斥锁更新、随机过期时间）。
3.  **向量索引参数**：HNSW 的 M 和 ef 参数对召回率和速度的具体影响是什么？
