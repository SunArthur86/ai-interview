# 嵌入模型用的什么?维度多少?怎么选的

**Situation：** 嵌入模型的选择直接影响向量检索质量.需要在效果、速度、成本之间找到平衡.

**Task：** 选择最适合中文企业场景的嵌入模型.

**Action：**
1. 候选模型对比:
   | 模型 | 维度 | 中文能力 | MTEB 排名 | 速度 |
   | :--- | :--- | :--- | :--- | :--- |
   | BGE-M3 | 1024 | 优秀 | 前 5 | 中 |
   | text-embedding-3-large | 3072 | 良好 | 前 3 | 快(API) |
   | BGE-large-zh | 1024 | 优秀 | 前 10 | 中 |
   | m3e-large | 1024 | 优秀 | 前 15 | 中 |
   | Cohere embed-v3 | 1024 | 良好 | 前 5 | 快(API) |

2. 评估维度:
   - **中文效果：** 使用内部评估集(300 条中文 query-doc pair),计算 Recall@10.
   - **多语言支持：** 部分文档是英文,需要跨语言检索能力.
   - **部署成本：** 本地部署 vs API 调用的成本对比.
   - **维度和存储：** 维度越高存储越大,检索越慢.

3. 最终选择 -- BGE-M3:
   - 中文检索 Recall@10 最高(94.3%).
   - 原生支持中英日韩多语言和跨语言检索.
   - 支持稠密向量、稀疏向量、多向量三种表示.
   - 1024 维在效果和效率间取得平衡. 可以本地部署,无数据出境风险.

4. 部署方案:
   - 使用 ONNX Runtime 加速推理.
   - GPU 部署(T4): 单条 embedding 延迟 15ms, batch 1000 条约 3s.
   - 预计算 + 增量更新: 新文档入库时实时计算 embedding.

**Result：**
- 中文检索 Recall@10 达到 94.3%,比 text-embedding-3-large 高 2.1%.
- 嵌入维度 1024,单条向量占 4KB,800 万文档的索引大小约 32GB.
- 本地部署避免了 API 调用成本和延迟.
