# Prompt Engineering 有哪些关键技巧

**回答：**

**1. 结构化 Prompt 设计：**
- 明确角色定义。
- 清晰的任务描述。
- 输出格式约束（如 JSON, Markdown）。
- 必要的限制条件（长度、字数、禁止词汇）。
- 高质量示例。

**2. 关键技巧：**
- **Chain of Thought (CoT)：** 引导模型逐步推理 ("请一步一步分析这个问题")。对于复杂逻辑任务，强迫模型输出思考链可显著提高准确率。
- **Few-shot Learning：** 提供 2-3 个高质量示例，模仿输入输出格式。示例的选取应覆盖边界情况，防止模型过拟合单一模式。
- **角色扮演：** "你是一个拥有 10 年经验的企业架构师"。利用模型训练数据中的角色知识库，激活特定领域的语调和思维模式。
- **否定指令：** "不要编造信息" 比 "请确保准确" 更有效。负向约束在某些模型中对减少幻觉效果更好。
- **输出锚定：** 在 Prompt 末尾给出输出的开头，引导格式。例如："Response: {"tool_name":"

**3. 高级技巧：**
- **Self-consistency：** 多次采样后投票，提高结果可靠性。适用于数学或逻辑推理题，通过多路径采样收敛出最一致的答案。
- **ReAct Prompting：** Thought-Action-Observation 循环，结合推理与工具调用。将推理过程与执行步骤解耦，适合 Agent 场景。
- **Instruction Hierarchy：** 系统指令 > 用户指令，防止 Prompt 注入。确保模型优先遵循开发者的安全或格式指令，而非用户的恶意越狱指令。

**4. 常见陷阱：**
- Prompt 过长导致注意力分散（关键信息被淹没）。
- 矛盾的指令导致模型困惑（如既要"简洁"又要"详细"）。
- 缺少格式示例导致输出不可解析（如 JSON 字段缺失）。

**## 常见考点**
1. **CoT 的局限性**：CoT 并不总是有效的，在模型本身不具备相关知识或对于简单事实性问题时，CoT 可能反而增加错误率。
2. **Few-shot vs Zero-shot**：何时优先使用 Few-shot？（通常用于格式对齐和复杂任务拆解），Zero-shot 优势在哪？（节省 Token，通用任务效果好）。
3. **提示词注入防御**：除了 Instruction Hierarchy，还有哪些手段？（如 Delimiters 分隔符 "###"，输出后处理校验）。
4. **Prompt 压缩**：面对长 Context 限制，如何优化 Prompt？（移除 Few-shot 示例，简化指令）。
