# 高并发下的限流降级策略是什么

**Situation：** 突发流量(如营销活动、批量查询)可能导致系统过载,需要保护核心服务不被打垮.

**Task：** 设计分级限流和优雅降级策略.

**Action：** 
1. **限流层次:**
   - **API 网关层:** 全局限流(总 QPS 上限 500).
   - **租户级限流：** 每个租户有独立的 QPS 配额(如 50 QPS).
   - **用户级限流：** 每个用户每分钟最多 20 次请求.

2. **限流算法:**
   - 使用 Redis + 令牌桶算法(Token Bucket).
   - 支持突发流量(桶容量 = 2倍的 QPS 配额).

3. **降级策略(分四个等级):**
| 等级 | 策略 |
| :--- | :--- |
| L0(正常) | 所有功能正常运行 |
| L1(轻微) | 关闭非核心功能(如推荐、个性化) |
| L2(中度) | 模型降级(复杂查询也走快速模型) |
| L3(严重) | 只返回缓存和FAQ回答,不调用LLM |

4. **自动降级触发条件:**
   - QPS > 阈值的 80% → L1.
   - LLM API 错误率 > 10% → L2.
   - 系统 CPU > 90% 或内存 > 85% → L3.

**Result：** 
- 在 5 倍突发流量下,核心功能仍然可用.
- 降级响应延迟 < 200ms(缓存/FAQ).
- 零系统崩溃事故.

## 常见考点
1. **限流算法对比**：为什么选择令牌桶而不是漏桶或固定窗口？令牌桶可以允许一定程度的突发流量（Burst），而漏桶强行平滑流量适合保护数据库。对于 API 层，令牌桶或滑动窗口更合适。
2. **热点 Key 问题**：在 Redis 限流中，如果某个特定 IP（如恶意攻击）请求量巨大，会不会打挂 Redis？需要考虑使用 Local Limit（本地限流，如 Guava RateLimiter）做第一道防线，或者对 Redis Key 进行分片。
3. **如何平滑恢复**：从 L3 降级恢复到 L0 的逻辑是什么？通常采用“阶梯式恢复”，如当 CPU < 70% 并持续 5 分钟后，才尝试恢复到 L2，避免系统在阈值附近反复震荡（系统抖动）。
