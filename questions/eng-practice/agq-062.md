# 怎么做的监控和告警

**Situation：** AI Agent 系统在生产环境中需要全方位监控：不仅是传统的系统指标，还包括 AI 特有的指标(幻觉率、工具调用成功率等)。
**Task：** 建立涵盖系统、业务、AI 三个维度的监控告警体系。
**Action：**
1.  **监控维度：**
    *   **系统层：** CPU、内存、磁盘、网络 I/O。Pod 状态、容器重启次数。数据库连接数、查询延迟。
    *   **业务层：** QPS、P50/P95/P99 延迟。错误率、超时率。用户活跃数、对话轮次分布。
    *   **AI 层(特有)：** LLM 调用延迟和成功率。Token 消耗趋势。检索命中率和 rerank 分数分布。幻觉率(基于 NLI 检测)。Agent 迭代次数分布(检测死循环趋势，若迭代次数 > N 则告警)。工具调用成功率和延迟。
2.  **监控工具栈：**
    *   **指标收集：** Prometheus (配置了 `/metrics` 抓取间隔为 15s)
    *   **可视化：** Grafana(定制了 AI Agent 专属 Dashboard)
    *   **日志：** ELK Stack(Elasticsearch + Logstash + Kibana)
    *   **链路追踪：** Jaeger (采用透传 TraceID 实现 Request Span 和 LLM Span 的关联)
    *   **AI 追踪：** LangFuse(记录每次 LLM 调用的输入输出)
3.  **告警规则：**
    *   **P1 (电话+钉钉)：** P99 延迟 > 10s，错误率 > 5% (持续 2 分钟)
    *   **P2 (钉钉)：** 幻觉率 > 5%，检索命中率 < 70%
    *   **P3 (邮件)：** Token 日消耗 > 200万
4.  **架构与数据流：**
```text
用户请求 (Agent Service)
   │
   ├─> 系统指标 (Node Exporter/ cAdvisor) ──> Prometheus ──> Alertmanager ──> 告警通道
   │
   ├─> 业务指标 (埋点 SDK) ─────────────────> Prometheus ──┘
   │
   └─> AI 上下文
         │
         ├─> 中间件/LLM Call (OpenTelemetry) ──> Jaeger ──> 链路分析
         │
         └─> 推理链路 ─────────────────────────> LangFuse ──> 可视化分析
```
**Result：**
问题发现时间从“用户投诉后”提前到“问题发生后 3 分钟”。
监控覆盖率 100%(所有核心链路都有指标)。
月均误报率 < 5%(告警阈值经过多次调优)。

## 常见考点
1. **Grafana 面板配置**：你们针对 AI Agent 设定了哪些核心 Key Performance Indicators (KPIs)？
2. **幻觉率检测**：在生产环境中，你们是如何自动化检测“幻觉”的？（追问：是用了模型判别还是关键词规则？）
3. **链路追踪**：在 LLM 多轮调用中，如何保证 TraceID 的透传一致性，特别是在涉及异步工具调用时？
4. **告警降噪**：面对 Agent 调用链路长导致的告警风暴，你们是如何做收敛或告警分级的？
