# 你的系统支持多租户吗?怎么实现的

**Situation：** 系统需要同时服务多个企业客户,每个客户有自己的知识库、配置和数据隔离要求.
**Task：** 设计多租户架构,确保数据隔离、配置独立、性能互不影响.
**Action：** 
1. **数据隔离策略(三层隔离)**:
   - **应用层(入口)：** 网关层通过 JWT 或 Header 解析 `tenant_id`，并透传给所有下游服务，确保上下文不丢失。利用 ThreadLocal 或 AsyncContext 在服务内部传递租户信息。
   - **数据层(存储)：**
     - **知识库隔离：** 每个租户在 Milvus 中使用独立的 Collection。在大数据量下， Partition 方案虽然物理上共用存储，但逻辑隔离，若数据量极不均衡（如某租户占 90% 数据）可能会影响查询性能，故采用独立 Collection 物理隔离，单租户故障不影响其他租户索引。
     - **对话数据隔离：** 每个租户的对话记录存储在独立的数据库 Schema 中（或 PostgreSQL 的 Schema 级别）。在 SQL 执行层面强制拼接 `WHERE tenant_id = ?`，利用数据库中间件（如 ShardingSphere）进行 SQL 拦截和重写，防止开发人员漏写过滤条件导致越权。
     - **缓存隔离：** Redis 中使用 `tenant_id:` 作为 key 前缀，并利用 Redis 的 DB 编号或 KeySpace 通知进行逻辑隔离，避免 Key 冲突。
2. **配置独立**:
   - **每个租户可以自定义：** System Prompt、工具集合、模型选择、检索策略。
   - 配置中心化存储，支持热更新。通过发布/订阅机制通知服务节点刷新本地缓存，确保无需重启服务。
3. **资源限制**:
   - **配额管理：** 每个租户有独立的 QPS 限制和 token 配额。
   - **限流算法：** 使用 Redis + Lua 脚本实现令牌桶或滑动窗口算法，保证分布式环境下限流的准确性。
   - **降级策略：** 超出配额时返回 429 Too Many Requests，并携带 Retry-After 头信息。

**系统架构图：**
```text
┌─────────────┐      ┌──────────────────────────────────┐
│   Client    │ ──▶  │       API Gateway (Auth)         │
└─────────────┘      │  - JWT Validation                │
                    │  - Extract tenant_id              │
                    └─────────────┬──────────────────────┘
                                  │ (Pass tenant_id)
                                  ▼
                    ┌──────────────────────────────────┐
                    │      Application Service          │
                    │  - ThreadLocal Context            │
                    │  - Rate Limiting Check            │
                    └─────┬───────────────┬──────────────┘
                          │               │
          ┌───────────────┴─────┐ ┌────────┴─────────────┐
          ▼                     ▼ ▼                      ▼
  ┌───────────────┐   ┌───────────────┐       ┌───────────────┐
  │  Milvus       │   │   Database    │       │    Redis      │
  │ (Collection)  │   │  (Schema)     │       │  (Key Prefix) │
  └───────────────┘   └───────────────┘       └───────────────┘
     Tenant A/B          Tenant A/B              Tenant A/B
```
**Result：** 
- 支持了 20+ 企业客户同时使用.
- 零数据泄露事故(严格的 tenant_id 过滤).
- 单个客户的异常不会影响其他客户(资源隔离).

## 常见考点
1. **隔离方案的权衡**：独立数据库 vs 共享数据库独立 Schema vs 共享共享表。各自的优缺点及适用场景？
2. **数据安全性**：如何防止开发人员在写 SQL 时忘记带 `tenant_id`？（SQL 拦截器 / ORM 生命周期钩子 / 数据库代理）
3. **性能影响**：大量租户下，连接池如何管理？是否需要针对不同租户做预热或隔离？
