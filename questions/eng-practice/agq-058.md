# 模型输出格式异常怎么处理

**Situation：** LLM 有时不按预期格式输出(如应输出 JSON 但输出了纯文本,或 ReAct 格式解析失败). 这种非确定性输出是 Agent 系统不稳定的主要来源之一。

**Task：** 建立鲁棒的输出解析和纠正机制，确保系统在模型输出不规范时仍能稳定运行。

**Action：** 
1. **多层解析策略:**
   采用「漏斗式」处理链路，逐步放宽解析条件，确保兜底：
   **严格正则解析 → 宽松正则解析 → LLM 重格式化 → 降级处理**

   ```text
   用户请求 → LLM 输出 ───────────────────────────────────────────────────────┐
                │                                                            │
                ▼                                                            │
   ┌──────────────────────┐  成功      ┌──────────────────────┐               │
   │  Layer 1: 严格正则   │───────────▶│ 返回结构化数据       │               │
   └──────────┬───────────┘            └──────────────────────┘               │
              │ 失败                                                    │
              ▼                                                          │
   ┌──────────────────────┐  成功      ┌──────────────────────┐               │
   │  Layer 2: 宽松正则   │───────────▶│ 返回结构化数据       │               │
   │ (容错: 空格/大小写)   │            └──────────────────────┘               │
   └──────────┬───────────┘                                                    │
              │ 失败                                                    │
              ▼                                                          │
   ┌──────────────────────┐  成功      ┌──────────────────────┐               │
   │  Layer 3: LLM 修复   │───────────▶│ 返回结构化数据       │               │
   │ (基于 Prompt 修正)   │            └──────────────────────┘               │
   └──────────┬───────────┘                                                    │
              │ 失败                                                    │
              ▼                                                          │
   ┌──────────────────────┐                                               │
   │  Layer 4: 降级处理   │◀───────────────────────────────────────────────┘
   │ (纯文本/默认回复)    │
   └──────────────────────┘
   ```

2. **具体实现细节:**
   - **第一层(严格正则):** 使用 `^\s*\{.*\}\s*$` (需包含多行模式匹配) 或 Pydantic 模型验证。针对 ReAct 严格匹配 `Thought:(.*)Action:(.*)`。
   - **第二层(宽松正则):** 使用 `json5` 或 `json-repair` 库，允许 trailing commas、注释、单引号等非标 JSON。
   - **第三层(LLM 修复):** 若前两层失败，提取原文本，注入 Prompt 如 "请修复以下文本中的 JSON 语法错误，仅返回 JSON:"，调用轻量模型（如 GPT-3.5/4o-mini）。
   - **第四层(降级):** 若 LLM 修复仍失败（如达到最大重试次数），将整个输出作为 Final Answer 直接返回，或触发人工介入流。

3. **预防措施:**
   - **Prompt Engineering:** 在 Prompt 中使用 XML 标签包裹输出（如 `<json_output>`），并增加 Few-shot Examples 展示标准格式。
   - **原生约束:** 使用 OpenAI 的 `response_format={"type": "json_object"}` 强制输出合法 JSON（但需配合 System Prompt 要求输出 JSON）。
   - **Tool Use:** 对于 Claude，优先使用 `tool_use` 模式替代纯文本 Prompt，模型对工具参数的生成准确率远高于纯文本生成。

4. **监控告警:**
   - **埋点:** 记录每一层解析失败的事件、原始输出片段和错误类型（如 syntax_error, missing_field）。
   - **阈值:** 格式遵循率 < 90% 时触发告警，自动回滚 Prompt 版本或检查模型版本稳定性。

**Result：** 
- 格式解析成功率从 82% 提升到 98%。
- LLM 修复层兜底了约 5% 的解析失败，延迟增加控制在 200ms 以内。
- 用户感知的格式异常（如看到原始 JSON 代码）从 8% 降低到 0.5%。

## 常见考点
1. **重试策略**：解析失败时直接重试还是先修复？如何避免 LLM 修复陷入死循环？（答：设置最大重试次数，如 2 次，超时则降级）。
2. **延迟权衡**：增加 LLM 修复层会增加延迟，如何优化？（答：使用更快的/更便宜的小模型进行修复，或者流式解析，部分字段解析失败时不阻塞整体展示）。
3. **JSON 模式局限**：OpenAI JSON Mode 无法保证包含特定字段怎么办？（答：需要在 Prompt 中显式列出必填字段，或在解析后补全默认值）。
