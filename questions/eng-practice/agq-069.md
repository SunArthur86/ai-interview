# 怎么做技术债务管理

**Situation：** 快速迭代中不可避免地产生技术债务:临时方案、代码坏味道、过时的依赖等.
**Task：** 建立技术债务的识别、记录和偿还机制.
**Action：** 
develop 冻结 → 创建 release 分支 → 集成测试 → 修复 bug
→ 合入 main → 打 tag → 部署

1. 技术债务识别:
Code Review 中标记技术债务(TODO: tech-debt  注释).
每次 Sprint 回顾时收集技术债务.
静态分析工具自动检测代码坏味道.
2. 分级管理:
P0(紧急): 影响稳定性或安全性 → 下个 Sprint 必须解决.
P1(重要): 影响可维护性或性能 → 1 个月内解决.
P2(一般): 代码风格、小优化 → 机会主义偿还.
3. 偿还策略:
每个 Sprint 预留 20% 的工时用于偿还技术债务.
大型重构制定专项计划.
**"童子军规则"：** 碰到的小债务随手修.
**Result：** 
技术债务积压量控制在可管理范围内.
系统可维护性评分持续 > 7 分(满分 10 分).
无因技术债务导致的严重事故.
