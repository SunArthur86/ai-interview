# 设计一个企业 RAG 系统的完整技术方案,从数据接入到用户查询的全流程是什么

- **企业 RAG 系统完整架构**

**数据流：** 数据源(PDF/Word/Web/DB) -> 数据处理管道(解析->分块->Embedding) -> 向量数据库(Milvus/Chroma)

**查询流：** 用户提问 -> Query改写 -> 多路检索(向量+BM25) -> RRF融合 -> Rerank -> LLM生成 -> 引用标注 -> 流式输出

- **Phase 1: 数据接入(离线)**
1. 文档解析:Unstructured/DeepDoc 处理 PDF(含表格/图片)
2. 清洗:去除页眉页脚、修复换行、合并段落
3. 分块:RecursiveCharacterSplitter(chunk=512, overlap=64)
4. 元数据标注:来源、页码、时间、权限标签

- **Phase 2: 检索(在线)**
1. Query 改写:HyDE / 子问题分解
2. 多路检索:向量检索 Top-20 + BM25 Top-20
3. RRF 融合:reciprocal_rank_fusion 合并
4. Rerank:Cross-Encoder 精排 Top-5
5. 上下文组装:父子块 + 窗口扩展

- **Phase 3: 生成(在线)**
1. Prompt 模板:System + Context + Question
2. LLM 生成:带引用标注的回答
3. 安全检查:幻觉检测 + 敏感词过滤
4. 流式输出:SSE 推送

- **## 常见考点**
1. **Rerank 的作用**：面试官必问。答案：向量检索是基于语义模糊匹配，召回率高但精度不够；Rerank（如 BGE-Reranker 模型）是 Cross-Encoder 结构，对 Query 和 Document 进行精细交互打分，能显著提升 Top-5 的相关性，但计算量大，不能用于全量召回。
2. **RRF (Reciprocal Rank Fusion)**：为什么需要融合算法？答案：向量检索擅长语义匹配（如“苹果公司”匹配“科技”），BM25 擅长关键词匹配（如“苹果”匹配“水果”），RRF 通过倒数排名融合两者，取长补短，鲁棒性更强。
3. **文档解析难点**：PDF 中的表格和图片如何处理？答案：表格使用 Table-Transformer 或 Markdown 转换，图片使用 OCR（如 PaddleOCR）或多模态模型（如 CLIP/Qwen-VL）提取文字/图注。
