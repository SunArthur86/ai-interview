# 如何设计一个支持多模型的 AI 网关?让客户可以根据场景切换不同的 LLM

- **多模型 AI 网关架构**

**请求流架构图：**

- **核心组件详解**

1.  **模型路由器**: 基于 Prompt Context 的动态决策
    *   **简单 FAQ (低 Token/高 QPS)**: GLM-4-Flash (便宜快速，甚至可以仅做 RAG 检索)
    *   **代码/逻辑推理 (高准确度)**: GPT-4 / Claude 3.5 Sonnet (代码能力强)
    *   **长文档阅读 (长上下文)**: Claude 3 (200k+ Context Window)
    *   **默认/通用**: GLM-4-Plus (性价比均衡)
    *   *实现细节*: 可通过 Prompt 预分类打标，或者基于 LlamaGuard 等轻量级模型进行意图识别。

2.  **统一适配层**: 屏蔽底层差异
    *   **字段映射**: 将 OpenAI 格式的 `messages` 映射到不同厂商的 `prompt`/`history` 结构。
    *   **流式转换**: 处理不同厂商 SSE (Server-Sent Events) 的数据分块格式差异。
    *   **参数归一化**: 统一 `temperature`, `top_p` 等参数范围。

3.  **智能降级链**:
    *   主模型 (GPT-4) [超时/限流/错误] -> 备用 -> 兜底 (GLM-4)
    *   *关键细节*: 需要实现熔断器模式，防止级联故障。

4.  **成本控制**:
    *   **Token 预算管理**: 租户级 Token 配额，超配额自动切换到廉价模型。
    *   **语义缓存**: 对 Query 进行向量相似度匹配，命中缓存直接返回 (命中率可达 20%-30%)。
    *   **模型降级**: 非核心业务在高峰期自动降级为低成本模型。

- **客户价值**: 一个 API 地址，背后自动选最优模型，降低成本同时保证效果。

## 常见考点
1.  **如何保证不同模型输出格式的一致性？**
    *   *追问*: 如果模型不遵循指定的 JSON 格式输出怎么办？
    *   *回答*: 使用 Pydantic 等库进行强类型校验，或者在 Adapter 层加入基于正则或 LLM 的后处理修正逻辑；使用 Function Calling 强制结构化输出。
2.  **语义缓存如何处理一致性问题？**
    *   *追问*: 缓存命中后，如果知识库更新了怎么办？
    *   *回答*: 设置合理的 TTL；或者实现版本化缓存，将文档 Hash 作为缓存 Key 的一部分。
3.  **多模型调用的超时控制策略？**
    *   *追问*: 有的模型响应要 10s，有的只要 1s，网关如何设置超时？
    *   *回答*: 网关层面的超时应设置为“业务可接受的延迟 + Buffer”，并记录各模型 P99 延迟动态调整路由权重。
