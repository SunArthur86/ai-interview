# 请设计一个日均 1000 万次调用的企业 AI 中台架构,月度成本控制在 50 万以内.

- **千万级日调用 AI 中台设计**

- **容量分析**
- 日 1000 万次调用 -> 平均 116 QPS,峰值 500+ QPS
- 假设每次调用平均 2000 token(1000 in + 1000 out)
- 日消耗 200 亿 token -> 月 6000 亿 token
- 纯 API 成本(GLM-4-Plus 约 0.05元/千token)= 月 300 万 -> 远超预算

- **成本优化策略(核心挑战)**

1. **模型分级路由(省 60%)**
- 70% 简单问题 -> GLM-4-Flash(0.001元/千token)
- 20% 中等问题 -> GLM-4-Air(0.01元/千token)
- 10% 复杂问题 -> GLM-4-Plus(0.05元/千token)
- 加权后约 0.008元/千token -> 月 48 万

2. **语义缓存(省 20-30%)**
- 高频相似问题直接返回缓存
- 缓存命中率 25% -> 再省 12 万/月

3. **自部署 Embedding(省 5%)**
- BGE-M3 本地部署,不再调用 Embedding API
- 节省约 2 万/月

4. **Prompt 精简(省 5%)**
- 上下文裁剪 + 历史摘要
- 减少 30% 输入 token

- **综合月成本**
- API:~36 万(路由+缓存优化后)
- 服务器:~8 万(4台 GPU 推理 + 6台 CPU 服务)
- 存储:~2 万(Milvus + Redis + PostgreSQL)
- 网络+CDN:~2 万
- 其他:~2 万
- **总计 ~50 万**

- **架构亮点**
- 模型路由器:用轻量分类器(FastText)预判问题复杂度
- 多级缓存:Redis(精确匹配)+ Milvus(语义匹配)
- 自动扩缩容:基于 QPS + Token 消耗率
- 成本告警:日消耗超预算 5% 自动通知

- **## 常见考点**
1. **路由策略实现**：模型路由的判断标准是什么？是基于 Prompt 长度、意图分类还是用户画像？（考察架构细节）
2. **缓存一致性**：当模型更新（如从 v1 升级到 v2）时，如何处理 Redis 和 Milvus 中的旧缓存？（考察数据一致性）
3. **冷启动问题**：系统初期 Milvus 没有向量数据，语义缓存如何发挥作用？（考察工程落地）
4. **成本突发控制**：如果某天因为攻击或 bug 导致 QPS 暴涨 10 倍，有什么熔断或降级机制保护预算？（考察稳定性设计）
